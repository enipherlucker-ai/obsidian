### 1. Модель выполнения: Интерпретатор, CPython и GIL (Global Interpreter Lock)

- **Что это?** Python — это интерпрерируемый язык. Ваш код выполняется не процессором напрямую, а специальной программой — **интерпретатором**. Самый популярный — **CPython** (написан на C).
    
- **GIL:** В CPython существует механизм **Global Interpreter Lock** — глобальная блокировка интерпретатора. Это мьютекс (блокировка), который позволяет в один момент времени выполняться только одному потоку (_thread_) кода на Python, даже на многоядерном процессоре.
    
- **Зачем это знать инженеру данных?**
    
    - **Многопоточность vs Многопроцессность:** GIL не мешает работать нескольким _процессам_ Python одновременно. Поэтому для CPU-bound задач (сжатие данных, сложные вычисления) нужно использовать модуль `multiprocessing`, а не `threading`. Для I/O-bound задач (сетевые запросы, чтение с диска) потоки (`threading`) подходят, так как во время ожидания I/O GIL отпускается.
        
    - **Производительность:** Это объясняет, почему "чистый" Python иногда медленнее компилируемых языков. Критичные к производительности участки кода выносят в C-расширения (как `numpy`, `pandas`) или используют несколько процессов.

### 2. Пространства имен (Namespaces) и область видимости (Scope)

- **Что это?** Пространство имен — это словарь, который сопоставляет имена переменных с объектами. Область видимости — это контекст, в котором это пространство имен доступно.
    
- **Порядок поиска (LEGB rule):** Интерпретатор ищет имя в следующем порядке:
    
    1. **L (Local):** Внутри текущей функции.
        
    2. **E (Enclosing):** В объемлющих функциях (замыкания).
        
    3. **G (Global):** На уровне модуля.
        
    4. **B (Built-in):** Во встроенных именах Python (например, `print`, `len`).
        
- **Зачем это знать инженеру данных?**
    
    - **Поиск багов:** Понимание, почему переменная внезапно `None` или имеет не то значение, часто упирается в область видимости.
        
    - **Замыкания (Closures):** Позволяют создавать функции с "памятью" (например, фабрики функций). Полезно для построения сложных конвейеров обработки.
        
    - **Избегание конфликтов имен:** Понимание, что глобальная переменная и переменная в функции — это разные вещи, если не объявлено `global`.

```python
x = "global" # Global

def outer():
    x = "enclosing" # Enclosing

    def inner():
        x = "local" # Local
        print(x)    # Найдет в Local -> "local"

    inner()

outer()
```

### 3. Протоколы и "Утиная типизация" (Duck Typing)

- **Что это?** Python сильно полагается на **протоколы** — неформальные интерфейсы. Если объект имеет нужные методы, он подходит для операции, даже если его класс не наследует от какого-то特定ного абстрактного класса.
    
    - _"Если что-то крякает как утка и плавает как утка, то это probably и есть утка."_
        
- **Примеры протоколов:**
    
    - **Итерабельный (Iterable):** объект, у которого есть метод `__iter__()` (или `__getitem__()`). Его можно использовать в цикле `for`.
        
    - **Итератор (Iterator):** объект с методом `__next__()`.
        
    - **Менеджер контекста (Context Manager):** объект с методами `__enter__()` и `__exit__()`. Используется с `with`.
        
- **Зачем это знать инженеру данных?**
    
    - **Гибкость кода:** Вы можете создавать свои классы, которые будут работать со стандартными конструкциями Python (`for`, `with`, `len()`), просто реализовав соответствующие методы (`__iter__`, `__enter__/__exit__`, `__len__`).
        
    - **Эффективная работа с данными:** Генераторы (функция с `yield`) — это легковесные итераторы. Они фундамент для обработки больших данных, не загружая их全部 в память. Вы постоянно используете их, даже если не замечаете (например, при `for line in open('huge_file.csv'):`).

```python
# Пример утиной типизации: всё, что имеет __len__, работает с len()
class MyFancyDataCollection:
    def __init__(self, data):
        self.data = data

    def __len__(self):  # Реализуем протокол "длины"
        return len(self.data) * 10

my_obj = MyFancyDataCollection([1, 2, 3])
print(len(my_obj))  # 30. Функция len() сработала!
```


### 4. Система импорта и структура проекта

- **Что это?** Как Python находит и загружает модули (`import my_module`).
    
- **Путь поиска (sys.path):** Список директорий, где Python ищет модули. Включает текущую директорию, стандартную библиотеку и пути из `PYTHONPATH`.
    
- **Разница между скриптом и модулем:** Если файл запускается напрямую (`python script.py`), его `__name__ == "__main__"`. Если импортируется, его `__name__` — это имя модуля.
    
- **Зачем это знать инженеру данных?**
    
    - **Организация кода:** Умение создавать пакеты (папки с `__init__.py`) — must-have для любого серьезного проекта, ETL-пайплайна или библиотеки утилит.
        
    - **Проблемы импорта:** Понимание, почему `ImportError` или импорт не того модуля — это часто проблема с `sys.path` или структурой проекта.
        
    - **Тестирование и переиспользование кода:** Код в блоке `if __name__ == "__main__":` не выполняется при импорте, что позволяет совмещать в одном файле исполняемый код и функции для импорта.

```text
my_data_project/
├── src/                       # Основной пакет
│   ├── __init__.py
│   ├── etl/                   # Подпакет
│   │   ├── __init__.py
│   │   ├── extract.py
│   │   └── transform.py
│   └── utils/
│       └── helpers.py
├── tests/                     # Тесты
├── scripts/                   # Исполняемые скрипты
│   └── run_pipeline.py
└── requirements.txt
```

### 5. Обработка исключений (Exception Handling)

- **Что это?** Механизм обработки ошибок и исключительных ситуаций.
    
- **Иерархия исключений:** Все исключения наследуются от `BaseException`. Важно ловить конкретные исключения, а не все подряд (`except Exception`).
    
- **Зачем это знать инженеру данных?**
    
    - **Отказоустойчивость пайплайнов:** Пайплайн обработки данных не должен падать из-за одной кривой строки в файле на миллион строк. Нужно уметь ловить исключения, логировать их и принимать решение (пропустить строку, остановить задание, отправить алерт).
        
    - **Контролируемое завершение ресурсов:** Менеджер контекста (`with open(...) as f:`) — лучший способ работы с ресурсами (файлы, сетевые подключения), так как гарантирует их закрытие даже при возникновении ошибки.

```python
try:
    df = pd.read_csv('input.csv', on_bad_lines='warn') # Парсим CSV
    result = complex_transformation(df)                 # Выполняем сложную логику
    result.to_parquet('output.parquet')                # Записываем результат
except FileNotFoundError:
    logging.error("Input file not found. Can't proceed.")
    send_alert("Pipeline failed!")
except pd.errors.EmptyDataError:
    logging.warning("File is empty. Creating an empty output.")
    create_empty_output()
except Exception as e: # Ловим ВСЁ остальное (не лучшая практика, но иногда нужна)
    logging.critical(f"Unexpected error: {e}", exc_info=True)
    send_alert("Pipeline failed mysteriously!")
```
