### **Конспект: Система типов в Python для инженера данных**

#### **1. Основной вопрос: Динамическая типизация — сила и слабость Python**

Python — язык с **сильной** и **динамической** типизацией. Это фундаментальное свойство, определяющее стиль разработки.

- **Динамическая типизация:** Тип переменной определяется в момент присваивания значения и может быть изменен.
    
```python
x = 42        # x сейчас int
x = "сорок два" # Теперь x это str. Ошибки нет.
x = [42]      # Теперь x это list. Интерпретатору всё равно.
```
- - **Плюс:** Гибкость и скорость написания кода, особенно в Jupyter Notebook для исследования данных.
        
    - **Минус:** Ошибки, связанные с несоответствием типа, всплывают только в **момент выполнения** (runtime). Код может упасть спустя долгое время после запуска.
        
- **Сильная типизация:** Язык не выполняет неявные преобразования между несовместимыми типами.
    
```python
# Попытка автоматически склеить строку и число приведет к ошибке
name = "Alice"
age = 30
message = name + " is " + age
# TypeError: can only concatenate str (not "int") to str

# Решение: явное преобразование
message = name + " is " + str(age)  # Корректно
```
- - **Плюс:** Код более предсказуем и менее подвержен скрытым багам из-за "магических" преобразований.
        
    - **Минус:** Требуется больше явных действий от разработчика.
        

**Вывод для инженера данных:** Мы работаем с "грязными" данными, где тип значения из CSV-файла или JSON-ответа API может быть неочевиден. Динамическая типизация позволяет быстро загрузить и исследовать данные, но сильная типизация тут же "бьет по рукам" при попытке сделать что-то несовместимое, защищая от некорректных операций.

#### **2. Эволюция подхода: Аннотации типов (Type Hints)**

Осознав недостатки "чистой" динамической типизации для больших проектов, сообщество Python (PEP 484) ввело **аннотации типов**. Это не меняет природу языка, а добавляет слой статического анализа.

**Цель:** Сделать код самодокументируемым и находить ошибки до запуска с помощью инструментов like **mypy** и **IDE** (PyCharm, VS Code).

**Базовый синтаксис аннотаций:**

```python

name: str = "Alice"
count: int = 100

# Для версий Python < 3.9 используется from typing import List -> tags: List[str]

# Аннотация функций (аргументов и возвращаемого значения)
def process_data(
    data: list[dict[str, int | str]],  # Список словарей с ключами-str и значениями int ИЛИ str
    threshold: int = 50
) -> pd.DataFrame:  # Функция обещает вернуть DataFrame

```
**Важно:** Интерпретатор Python **игнорирует** аннотации типов. Код выполняется как обычно. Аннотации — это подсказки для разработчика и инструментов анализа.

**Практическая польза для инженера данных:**

1. **Автодополнение в IDE:** Зная тип переменной (`pd.DataFrame`, `list`), среда разработки сможет подсказать все доступные методы (`.groupby()`, `.append()`).
    
2. **Обнаружение ошибок ДАТАФРАЙМА:** Самая частая ошибка — опечатка в названии колонки.
    
```python
def clean_data(df: pd.DataFrame) -> pd.DataFrame:

```
**Валидация конфигов и данных:** Аннотации помогают описывать ожидаемую структуру данных.

```python

def init_pipeline(config: DataConfig) -> None:

```

#### **3. Дополнительные вопросы и ответы**

**Вопрос 1: Обязательно ли использовать аннотации типов в Python-проектах?**  
**Ответ:** Нет, не обязательно для запуска кода. Но для инженера данных уровня **middle** это обязательный навык и признак качественного кода. В современных коммерческих проектах использование аннотаций и линтеров (mypy) — это стандарт де-факто.

**Вопрос 2: Как аннотировать сложные структуры, специфичные для Data Engineering?**  
**Ответ:**

- **Pandas:** `pd.DataFrame`, `pd.Series`. Для более сложных случаев можно указать типы колонок с помощью `pandas.DataFrame[{"col1": pd.Series[dtype=int], ...}]`, но это сложно и часто избыточно. Обычно достаточно `-> pd.DataFrame`.
    
- **Опциональные значения:** `from typing import Optional`. `user_id: Optional[int] = None` означает, что `user_id` может быть `int` или `None`.
    
- **Любой из типов:** `int | str` (Python 3.10+) или `Union[int, str]`. Полезно для колонкок, которые могут содержать и числа, и строки.
    
- **Классы данных (dataclasses):** Идеальны для описания конфигураций или моделей данных.
    
```python
from dataclasses import dataclass

@dataclass
class ETLTask:
    source_uri: str
    target_table: str
    batch_size: int = 1000
    drop_table: bool = False
```
**Вопрос 3: Как начать использовать аннотации в существующем проекте?**  
**Ответ:** Постепенно. Сначала аннотировать новые функции и модули. Затем — медленно рефакторить старый код. Запускать `mypy` можно сначала в нестрогом режиме, отключая проверки для частей кода без аннотаций (`# type: ignore`).

#### **4. Особенности и "подводные камни"**

1. **`Any` — аварийный люк:** Тип `Any` отключает проверку типов. Его использование — это признак того, что вы не можете или не хотите описать тип. **Избегайте `Any`**. Лучше использовать более общие типы (`Union`, `Protocol`) или пересмотреть архитектуру.
    
```python
from typing import Any
def risky_function(data: Any) -> Any:  # mypy не будет проверять ничего
    return data * 2  # Очень опасно!
```
- **Производительность:** Аннотации типов — это overhead времени разработки, а не времени выполнения. На скорость работы программы они **никак не влияют**, так как являются лишь метаданными.
    
- **Дисциплина и соглашения:** Самая большая сложность — договориться в команде об уровне строгости проверок и consistently добавлять аннотации. Без этого их польза резко падает.
    
- **Итог для Middle-инженера:** Владение системой типов Python — это не про знание `typing.py`, а про **понимание компромисса**. Используйте динамическую типизацию для быстрого прототипирования и исследования, но обязательно укрепляйте критические участки пайплайнов (функции преобразования данных, конфиги, публичные API) аннотациями типов. Это прямой путь к созданию надежного, поддерживаемого и понятного кода, что и отличает senior-подход от junior.
