#### **Что такое "Куча" (Heap)**

**Контекст: Классические СУБД (например, PostgreSQL)**

- **"Куча" (Heap Table)** — это таблица, хранящая данные в неупорядоченном виде.
- **Движок:** В PostgreSQL по умолчанию используется `HEAP`.
- **Принцип:** Новые строки просто добавляются в первое доступное место. Нет предопределенного порядка.
- **Для поиска** по такому хранилищу обязательно требуются **индексы**. Без индекса для поиска одной строки потребуется полное сканирование таблицы (FULL SCAN).
    

**ClickHouse: Уходим от "Кучи"**  
В ClickHouse **нет понятия "кучи"** в том виде, как в OLTP-системах. Это осознанный отказ от неупорядоченного хранения ради скорости аналитических запросов.

**Основные движки семейства `*MergeTree`** (de facto стандарт):

- **`MergeTree`:** Основной движок. Данные **всегда физически сортируются** по ключу (`ORDER BY`).
- **Принцип:** Данные разбиваются на куски (parts), и внутри каждого куска данные отсортированы по ключу. Это позволяет:
    1. **Эффективно использовать индексы:** Разреженный индекс (primary.idx) быстро находит нужные куски данных.
    2. **Выполнять быструю агрегацию:** Данные, сгруппированные по нужным полям, уже лежат рядом.
    3. **Сжимать данные:** Упорядоченные данные сжимаются гораздо эффективнее.

**Другие движки ClickHouse:**

- **`Log`/`TinyLog`/`StripeLog`:** Простейшие движки для небольших таблиц (до ~1 млн строк). Данные не сортируются (аналог "кучи"), но не предназначены для серьезной аналитики.
- **`ReplacingMergeTree`:** Удаляет дубликаты по ключу сортировки _в фоне_ при слиянии кусков.
- **`SummingMergeTree`:** Агрегирует данные по ключу сортировки при слиянии.
- **`ReplicatedMergeTree`:** Надстройка над `MergeTree` для обеспечения репликации данных across кластер.
