#### **1. Сущность и определение**

**Внешний ключ (Foreign Key, FK)** — это ограничение (`CONSTRAINT`) в [[Непонятные слова#Реляционная СУБД|реляционной СУБД]], которое обеспечивает **ссылочную целостность** данных между двумя таблицами.

- **Механизм:** Поле (или набор полей) в одной таблице (`дочерней`) ссылается на первичный ключ (`PRIMARY KEY`) или уникальное ограничение (`UNIQUE`) в другой таблице (`родительской`).

- **Цель:**
    1. **Гарантировать целостность данных:** Нельзя вставить в дочернюю таблицу запись с несуществующим в родительской таблице значением ключа.
    2. **Предотвратить каскадные удаления:** Нельзя удалить запись из родительской таблицы, если на нее есть ссылки в дочерней (зависит от настроек).
    3. **Явно декларировать связь** между сущностями в схеме данных.

---

#### **2. Реализация в классических реляционных СУБД (OLTP)**

В СУБД типа PostgreSQL, MySQL, Oracle внешние ключи — это краеугольный камень модели данных.

- **Синтаксис (на примере PostgreSQL):**
    
 ```    sql
CREATE TABLE department (
    id INT PRIMARY KEY,
    name VARCHAR(255)
);

CREATE TABLE employee (
    id INT PRIMARY KEY,
    name VARCHAR(255),
    department_id INT REFERENCES department(id) -- Простейший вариант
    -- Или с явным объявлением CONSTRAINT
    -- CONSTRAINT fk_employee_department
    --     FOREIGN KEY(department_id)
    --     REFERENCES department(id)
    --     ON DELETE CASCADE -- Опция действия при удалении
);
    ```
    
- **Опции при удалении/обновлении (`ON DELETE` / `ON UPDATE`):**
    - `RESTRICT` (по умолчанию): Запретить удаление родительской записи, если есть ссылки.
    - `CASCADE`: Автоматически удалить все дочерние записи при удалении родительской.
    - `SET NULL`: Установить значение внешнего ключа в `NULL` у всех дочерних записей.
    - `SET DEFAULT`: Установить значение внешнего ключа в значение по умолчанию.
- **Плюсы:** Полная гарантия целостности данных на уровне БД.
- **Минусы:** Накладные расходы на проверку ограничений при каждой операции `INSERT`/`UPDATE`/`DELETE`, что может снижать производительность при массовых загрузках.

#### **3. Особенность ClickHouse: Отсутствие настоящих внешних ключей**

**Важно!** ClickHouse — это **OLAP-система**, ориентированная на скорость агрегации и вставки больших пакетов данных. Поддержка классических ограничений целостности, таких как внешние ключи, **отсутствует**.

- **Причины:**
    1. **Производительность:** Проверка ограничений для каждой вставляемой строки свела бы на нет главное преимущество ClickHouse — скорость.
    2. **Массовые вставки:** Данные загружаются большими пачками (`INSERT ... SELECT`), и проверка целостности для миллионов строк была бы крайне дорогой.
    3. **Архитектура:** Отсутствие полноценных транзакций на уровне всего кластера делает реализацию согласованных FK практически невозможной.

#### **4. Эмуляция связей в ClickHouse: Как инженеру данных работать без FK**

Поскольку БД не гарантирует целостность, ответственность за нее переносится на процессы ETL/ELT и архитектуру данных.

| Подход / Объект                          | Описание                                                                                                                                                                   | Плюсы                                                                                    | Минусы                                                                              | Когда использовать                                                                 |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| **Словари (Dictionaries)**               | **Наиболее близкая альтернатива.** Позволяют загрузить справочную (родительскую) таблицу в память и подставлять значения по ключу "на лету" с помощью функции `dictGet()`. | • Высокая скорость доступа.  <br>• Интегрированы в CH.  <br>• Можно обновлять.           | • Требуют памяти.  <br>• Не enforce целостность, это лишь удобный способ JOIN'а.    | Для частых подстановок значений из справочников (например, название товара по ID). |
| **Ссылочные таблицы (Reference Tables)** | Небольшаяdimension-таблица, которая **реплицируется** на все ноды кластера. При JOIN'е к ней копия таблицы есть на каждой ноде, что делает join эффективным.               | • Эффективные JOIN'ы.  <br>• Поддерживает целостность "вручную".                         | • Размер ограничен (маленькие таблицы).  <br>• Данные могут устареть до репликации. | Справочники, валюты, категории — данные, которые редко меняются.                   |
| **Объединение таблиц (JOIN)**            | Выполнение JOIN'ов в запросах. ClickHouse умеет в разные виды JOIN'ов, но их нужно использовать аккуратно.                                                                 | • Гибкость.  <br>• Не требует подготовки.                                                | • **Может быть очень медленным** при больших right-таблицах.                        | Для ad-hoc-аналитики, но не для высокопроизводительных пайплайнов.                 |
| **Денормализация**                       | **Основной метод в OLAP.** Предварительное соединение данных из разных сущностей в одну широкую таблицу на этапе ETL.                                                      | • Максимальная скорость выполнения запросов.  <br>• Идеально для фильтрации и агрегации. | • Избыточность данных (больший объем).  <br>• Сложность обновления.                 | Частая необходимость фильтровать/агрегировать по атрибутам "чужой" сущности.       |

**Вывод для ClickHouse:** Инженер данных должен **спроектировать процесс загрузки данных (ETL) так, чтобы он гарантировал целостность** перед попаданием в ClickHouse. В самой СУБД используются **словари и денормализация** для эффективной работы со связями.

---

#### **5. Роль инженера данных в обеспечении целостности**

1. **Валидация на этапе ETL/ELT:** Проверка существования ключей должна происходить в пайплайне загрузки (с помощью скриптов Python, в dbt-тестах и т.д.).
2. **Согласованные срезы данных:** При загрузке данных из операционных систем (OLTP) необходимо использовать **технику снимков (snapshots)** или логирования, чтобы избежать ситуации, когда в ClickHouse попадает запись из дочерней таблицы, а ссылочная запись из родительской — еще нет.
3. **Архитектура хранения:** Частое использование **широких фактовых таблиц**, где все необходимые атрибуты уже проставлены (денормализация), чтобы избежать необходимости JOIN'ов в запросах.

---

#### **6. Вывод для инженера данных уровня Middle**

1. **Классические FK** — это инструмент **OLTP-систем** для гарантии целостности на уровне БД. Вы должны знать их синтаксис и поведение.
2. **В ClickHouse и других OLAP-системах FK отсутствуют.** Это **осознанный trade-off** в пользу производительности при работе с большими данными.
3. **Ответственность за целостность данных** переносится с уровня БД на уровень **процессов инженерии данных (ETL/ELT)** и архитектуры.
4. **Основные инструменты работы со связями в ClickHouse:** **денормализация**, **словари** и реплицируемые **справочные таблицы**. JOIN используется как крайняя мера для ад-hoc-аналитики.

Понимание этой фундаментальной разницы между OLTP и OLAP и умение компенсировать отсутствие FK другими методами — ключевой навык для инженера данных, работающего с большими данями.
