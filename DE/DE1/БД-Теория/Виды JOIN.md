## Основные виды JOIN

| Вид JOIN                     | Схема объединения            | Краткое описание                                                                            | Пример применения                                  |
| ---------------------------- | ---------------------------- | ------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| **INNER JOIN**               | Пересечение                  | Возвращает только совпадающие записи из обеих таблиц                                        | Список заказов, у которых есть клиенты             |
| **LEFT JOIN (LEFT OUTER)**   | Левая таблица + пересечение  | Все записи из левой таблицы + совпадения из правой (NULL, если нет совпадений)              | Все клиенты, даже без заказов                      |
| **RIGHT JOIN (RIGHT OUTER)** | Правая таблица + пересечение | Все записи из правой таблицы + совпадения из левой (NULL, если нет совпадений)              | Все товары, даже если не продавались               |
| **FULL JOIN (FULL OUTER)**   | Объединение                  | Все записи из обеих таблиц, совпадения показываются один раз, несовпадения дополняются NULL | Все клиенты и все заказы, даже если они не связаны |
| **CROSS JOIN**               | Декартово произведение       | Каждая строка левой таблицы соединяется с каждой строкой правой                             | Все комбинации товаров и магазинов                 |
| **SELF JOIN**                | Таблица сама с собой         | Используется для иерархий или поиска связей внутри таблицы                                  | Дерево сотрудников: кто чей руководитель           |
## 1. Общая идея

JOIN — это операция объединения строк из двух таблиц по какому-то условию (обычно `ON`).  
Алгоритм выполнения JOIN зависит от СУБД, размеров данных, наличия индексов и доступной памяти.

---

## 2. Основные алгоритмы JOIN

### 🔹 2.1. Nested Loop Join

- **Принцип**: Для каждой строки из одной таблицы ищем подходящие строки во второй.
- **Сложность**: O(N × M) в худшем случае.
- **Когда используется**:
    - Маленькая таблица соединяется с большой, и на большой есть индекс по ключу JOIN.
- **Пример** (PostgreSQL):
```sql
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id;
```
 Если `customers.id` индексирован, то для каждой строки `orders` будет быстрый поиск в `customers`.
- **Подводный камень**: На больших таблицах без индексов — **дико медленно**.

---

### 🔹 2.2. Merge Join (Sort-Merge Join)

- **Принцип**: Обе таблицы сортируются по ключу, затем сканируются «слева направо» и сливаются.
- **Сложность**: O(N log N + M log M), т.е. зависит от сортировки.
- **Когда используется**:
    - Когда данные уже отсортированы (например, по кластерному индексу).
    - Когда обе стороны большие, и индексы не помогут.
- **Пример**:  
    В PostgreSQL оптимизатор может выбрать merge join, если есть `ORDER BY` или данные в индексе B-Tree.
- **Подводный камень**: сортировка может быть дорогой по CPU и памяти.

---

### 🔹 2.3. Hash Join

- **Принцип**:
    1. Выбирается меньшая таблица, по ключу строится **хэш-таблица**.
    2. По строкам из второй таблицы ищем совпадения в хэш-таблице.
- **Сложность**: O(N + M).
- **Когда используется**:
    - Наиболее распространённый метод в современных СУБД (Postgres, ClickHouse).
    - Хорошо работает, если есть достаточно памяти.
- **Пример** (ClickHouse):
```sql
SELECT *
FROM orders AS o
ANY INNER JOIN customers AS c
ON o.customer_id = c.id;
```

- ClickHouse построит хэш-таблицу по `customers` и будет искать по ключу `customer_id`.
- **Подводные камни**:
    - Если хэш-таблица не помещается в память → сброс на диск (деградация производительности).
    - Нужно следить за параметрами памяти (`max_bytes_in_join`, `join_algorithm`).

---

### 🔹 2.4. Grace Hash Join

- **Принцип**: Расширение обычного hash join для очень больших таблиц, которые не помещаются в память.  
    Делает разбиение (partitioning) на чанки и строит хэши по кускам.
- **Сложность**: ближе к O(N + M), но с дополнительными затратами на запись/чтение с диска.
- **Использование**:
    - PostgreSQL и некоторые аналитические СУБД.
    - В ClickHouse чаще стараются увеличить память и использовать обычный hash join.

---

### 🔹 2.5. Adaptive Join (современный подход)

- **Принцип**: СУБД может **динамически выбрать** алгоритм (hash / nested loop / merge) в процессе выполнения, если статистика оказалась неточной.
- **Применение**:
    - SQL Server, Oracle, PostgreSQL (частично через JIT), в ClickHouse чаще явно указывают алгоритм (`SET join_algorithm`).
- **Подводный камень**: Не всегда предсказуемое поведение.

---

## 3. Особенности в ClickHouse и PostgreSQL

### В ClickHouse

- Поддерживаются **hash join**, **merge join** (частично) и **direct join** (lookup по ключу).
- Управление через параметры:
    - `join_algorithm = 'auto' | 'hash' | 'merge' | 'partial_merge'`
    - `max_bytes_in_join`, `max_rows_in_join`.
- Для больших таблиц часто используют `ANY LEFT JOIN`, чтобы ограничить количество строк (экономит память).

### В PostgreSQL

- Есть **nested loop**, **merge join**, **hash join** (и их гибриды).
- Оптимизатор сам выбирает алгоритм на основе статистики.
- Хэш-таблицы могут сбрасываться на диск, если не хватает `work_mem`.

## 4. Возможные дополнительные вопросы

**Q1. Чем отличается Nested Loop от Hash Join?**

- Nested Loop перебирает строки (медленно на больших объёмах), Hash Join строит хэш-таблицу (быстрее, но требует памяти).

**Q2. Почему hash join может деградировать?**

- Если хэш-таблица не помещается в RAM, СУБД начнёт писать на диск → резкий рост времени.

**Q3. Как в ClickHouse управлять выбором алгоритма join?**

- Через настройку `SET join_algorithm = 'hash'` или `merge`. Обычно `auto` выбирает оптимальный вариант.

**Q4. Можно ли ускорить merge join?**

- Да, если данные уже отсортированы по ключу, сортировка будет пропущена.

**Q5. Что выбрать для маленькой таблицы и большой?**

- Hash Join или Nested Loop с индексом (если в Postgres). В ClickHouse почти всегда hash join.

---

## 5. Подводные камни

- В ClickHouse **join сильно зависит от памяти** — можно случайно «убить» сервер большим JOIN.
- В Postgres важно правильно настраивать `work_mem`, иначе hash join уйдёт на диск.
- JOIN в ClickHouse работает **только по равенству** (`=`), нет поддержки `>` или `BETWEEN`.
- Если ключи не уникальны, количество строк может сильно увеличиться (Cartesian product).

---

✅ **Итог**:  
Основные алгоритмы — Nested Loop, Merge Join, Hash Join (+ Grace Hash для больших данных).  
В реальной практике (ClickHouse, PG) чаще всего используется **Hash Join** как наиболее эффективный.  
Нужно учитывать **размеры таблиц**, **память** и **индексы**, чтобы JOIN не стал «бутылочным горлышком».
