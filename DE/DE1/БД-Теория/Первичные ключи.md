#### **1. Классическое определение (OLTP: PostgreSQL, MySQL, etc.)**

**Первичный ключ (Primary Key, PK)** — это ограничение целостности (`CONSTRAINT`), которое **однозначно идентифицирует каждую запись** в таблице реляционной СУБД.

- **Обязательные свойства:**
    1. **Уникальность:** В таблице не может быть двух строк с одинаковым значением первичного ключа.
    2. **Не-null:** Ни одна часть ключа не может содержать значение `NULL`.
    3. **Неизменность:** Значение ключа не должно изменяться со временем (или изменяться крайне редко).
- **Роль и функции:**
    - **Гарантия целостности:** СУБД жестко контролирует уникальность при каждой операции `INSERT`/`UPDATE`.
    - **Обеспечение связей:** Является целью для ссылок со стороны **внешних ключей (FOREIGN KEY)** в связанных таблицах.
    - **Создание индекса:** При создании PK автоматически создается уникальный индекс (часто кластерный), который обеспечивает быстрый поиск по ключу.
- **Синтаксис (пример на PostgreSQL):**
```sql
CREATE TABLE users (
    user_id BIGSERIAL PRIMARY KEY, -- Автоинкрементный PK
    email VARCHAR(255) NOT NULL UNIQUE
);
```
**Вывод:** В OLTP-системах PK — это, в первую очередь, **гарант целостности данных** и основа для связей между таблицами.

#### **2. Первичный ключ в ClickHouse: фундаментальные различия**

ClickHouse — это OLAP-система, ориентированная на скорость обработки больших объемов данных. Его архитектура в корне меняет семантику первичного ключа.

|**Аспект**|**Классические СУБД (OLTP)**|**ClickHouse (OLAP)**|
|---|---|---|
|**Основная цель**|Гарантия уникальности записи и целостности данных.|**Оптимизация запросов.** Определяет, как построен **разреженный индекс** для быстрого поиска данных.|
|**Гарантия уникальности**|**Да, абсолютная.** СУБД отвергает нарушающие вставки.|**Нет.** ClickHouse **не проверяет уникальность** при вставке. В таблице могут быть дубликаты по PK.|
|**Связь с данными**|Логическое ограничение.|Физическая организация данных. Неразрывно связан с **ключом сортировки (`ORDER BY`)**.|
|**Состав ключа**|Часто технический `ID` (суррогатный ключ).|**Всегда составной и семантический.** Включает поля, по которым часто фильтруют и агрегируют (например, `(date, user_id)`).|

**Золотое правило ClickHouse:** Первичный ключ (`PRIMARY KEY`) — это, по умолчанию, **префикс** ключа сортировки (`ORDER BY`). Если `ORDER BY (a, b, c)`, то `PRIMARY KEY (a, b)` будет валидным, а `PRIMARY KEY (b, c)` — нет.

#### **3. Как работает первичный ключ в ClickHouse?**

Механизм основан на **разреженном индексе** (sparse index), который позволяет работать с огромными таблицами, не загружая индекс целиком в память.

1. **Данные физически сортируются** по ключу `ORDER BY` (который включает PK) и разбиваются на куски (parts).
    
2. Для каждого куска в **первичном индексе** (`primary.idx`) сохраняется только **первая запись** каждого _индексного гранула_ (по умолчанию каждые 8192 строки).
    
3. При запросе с `WHERE` по полям из PK:
    
    - ClickHouse смотрит в небольшой файл `primary.idx`.
        
    - Быстро определяет, **какие куски (parts) и гранулы внутри них** _могут_ содержать искомые данные.
        
    - Сканирует только эти гранулы, а не всю таблицу.
        

**Пример:**
```sql
CREATE TABLE sales (
    event_date Date,
    product_id UInt32,
    user_id UInt32,
    revenue Float64
) ENGINE = MergeTree
ORDER BY (event_date, product_id, user_id) -- Ключ сортировки
PRIMARY KEY (event_date, product_id) -- Первичный ключ (префикс ORDER BY)

-- Запрос 1: Быстрый (использует PK)
SELECT sum(revenue) FROM sales
WHERE event_date = today() AND product_id = 123;

-- Запрос 2: Медленный (полное сканирование, т.к. user_id не в PK)
SELECT sum(revenue) FROM sales WHERE user_id = 456;
```

#### **4. Рекомендации по выбору PK в ClickHouse для инженера данных**

1. **Принцип селективности:** Располагайте поля в PK от **наименее селективного** к **наиболее селективному**.
    - **Плохо:** `PRIMARY KEY (user_id, date)` – `user_id` очень селективен, индекс неэффективно отсечет данные.
    - **Хорошо:** `PRIMARY KEY (date, user_id)` – сначала отсекаем целые дни, затем ищем внутри дня по `user_id`.

2. **Включайте в PK поля, часто используемые в:**
    - `WHERE` для фильтрации.
    - `GROUP BY` для агрегации.
    - `JOIN` по условию.

3. **Избегайте включения в PK полей с очень высокой селективностью** (как единственный элемент), например, `UUID`. Это приведет к тому, что индекс "разрастется" и будет неэффективен. Для таких сценариев лучше использовать **skipping indexes**.

4. **Используйте функциональные возможности.** Ключ может быть выражением:
```sql
ORDER BY (toStartOfHour(event_time), user_id) -- Сортировка по часу
```

#### **5. Если нужна уникальность? Альтернативы в ClickHouse**

Поскольку PK не обеспечивает уникальность, инженер данных должен выбрать другой инструмент:

1. **Денормализация:** Сделать одну широкую таблицу, где строки уникальны по составу данных.
2. **Движок `ReplacingMergeTree`:** Позволяет задать версию строки (`version`). Позволяет оставить только последнюю версию строки с одинаковым ключом сортировки _в процессе фонового слияния кусков_. **Важно:** Дедупликация не моментальна и не гарантирована на 100% в любой момент времени.
3. **Движок `CollapsingMergeTree`/"Оперативный" `GROUP BY`:** Для данных, которые можно "схлопнуть" по знаку.
4. **Гарантия на уровне ETL:** Обеспечить уникальность данных на этапе загрузки в ClickHouse.

---

#### **6. Итоговый вывод для инженера данных Middle**

1. **Забудьте о классическом PK:** В ClickHouse первичный ключ — это **не про уникальность**, а про **физическую организацию данных** и **индексирование**.
2. **`ORDER BY` важнее `PRIMARY KEY`:** Ключ сортировки — главный элемент проектирования таблицы. PK — лишь его префикс для индекса.
3. **Ваша задача — проектирование под запросы:** Выбор полей для PK — это ключевое решение, определяющее производительность сотен запросов. Вы должны анализировать паттерны доступа к данным.
4. **Уникальность — ваша ответственность:** ClickHouse не обеспечивает ее из коробки. Вы должны выбрать стратегию (движок, ETL-процесс) для ее достижения, если это требуется.

Понимание этой разницы и умение правильно применить первичный ключ в ClickHouse — один из ключевых навыков, отличающий Junior от Middle-инженера данных.
