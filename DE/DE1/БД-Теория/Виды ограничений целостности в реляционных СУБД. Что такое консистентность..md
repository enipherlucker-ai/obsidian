**Основная цель:** Обеспечить корректность, непротиворечивость и надежность данных в базе, предотвращая появление "мусора" и логических ошибок. Это основа для достоверной аналитики.

---

### **1. Основные концепции с примерами**

#### **Что такое консистентность (согласованность)?**

**Консистентность** — это состояние, при котором данные в базе соответствуют всем заданным правилам, ограничениям и бизнес-логике. Это результат применения **ограничений целостности (integrity constraints)**.

Простыми словами: **Консистентность — это цель, а ограничения целостности — инструменты для ее достижения.**

#### **1. Ограничения целостности сущности (Entity Integrity)**

- **Цель:** Гарантировать, что каждая строка (сущность) в таблице может быть uniquely identified.
    
- **Инструмент:** `PRIMARY KEY` (Первичный ключ).
    
- **Правила:**
    
    1. Уникальность: Значение первичного ключа не может повторяться.
        
    2. Не NULL: Значение первичного ключа не может быть пустым.
        
- **Пример:**
```sql
CREATE TABLE dim_product (
    product_id INT PRIMARY KEY, -- Ограничение целостности сущности
    product_name VARCHAR(100) NOT NULL,
    sku VARCHAR(50) UNIQUE
);
```

- - `INSERT INTO dim_product VALUES (1, 'Phone', 'SKU123');` -- OK
        
    - `INSERT INTO dim_product VALUES (1, 'Laptop', 'SKU456');` -- ОШИБКА: нарушение уникальности PK
        
    - `INSERT INTO dim_product VALUES (NULL, 'Tablet', 'SKU789');` -- ОШИБКА: PK не может быть NULL
        

#### **2. Ограничения целостности ссылок (Referential Integrity)**

- **Цель:** Гарантировать, что связи между таблицами не будут нарушены. Ссылка на другую сущность должна существовать.
    
- **Инструмент:** `FOREIGN KEY` (Внешний ключ).
    
- **Правила:** Значение во внешнем ключе должно либо быть NULL (если разрешено), либо совпадать с одним из значений первичного ключа в связанной (родительской) таблице.
    
- **Пример:**
```sql
CREATE TABLE fact_sales (
    sale_id INT PRIMARY KEY,
    product_id INT NOT NULL,
    amount DECIMAL(10,2),
    -- Ограничение целостности ссылок:
    CONSTRAINT fk_sales_product
      FOREIGN KEY (product_id)
      REFERENCES dim_product(product_id)
      ON DELETE RESTRICT -- Вариант действия при удалении
);
```

- - `INSERT INTO fact_sales VALUES (100, 1, 999.99);` -- OK, товар с ID=1 существует.
        
    - `INSERT INTO fact_sales VALUES (101, 999, 499.99);` -- ОШИБКА: товара с ID=999 нет в dim_product.
        

#### **3. Ограничения целостности домена (Domain Integrity)**

- **Цель:** Гарантировать, что значения в столбце соответствуют заданному типу данных, формату или диапазону.
    
- **Инструменты:** `DATA TYPE`, `NOT NULL`, `UNIQUE`, `CHECK`, `DEFAULT`.
    
- **Примеры:**
```sql
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(100) NOT NULL, -- Ограничение домена: не пустое
    email VARCHAR(255) UNIQUE,      -- Ограничение домена: уникальное
    age INT CHECK (age >= 18),      -- Ограничение домена: проверка
    department VARCHAR(50) DEFAULT 'Unassigned', -- Ограничение домена: значение по умолчанию
    salary DECIMAL(10,2) CHECK (salary > 0) -- Ограничение домена: проверка
);
```

- - `INSERT INTO employees (emp_id, emp_name, age, salary) VALUES (1, 'John', 17, 50000);` -- ОШИБКА: `age < 18`
        
    - `INSERT INTO employees (emp_id, emp_name, age, salary) VALUES (2, 'Jane', 25, -1000);` -- ОШИБКА: `salary < 0`
        

#### **4. Ограничения целостности пользователя (User-Defined Integrity)**

- **Цель:** Реализация сложной бизнес-логики, которую нельзя описать стандартными ограничениями.
    
- **Инструменты:** `CHECK` для сложных условий, `TRIGGERS` (триггеры), `STORED PROCEDURES` (хранимые процедуры).
    
- **Пример:**

```sql
CREATE TABLE bookings (
    booking_id INT PRIMARY KEY,
    check_in_date DATE,
    check_out_date DATE,
    -- Сложная бизнес-логика: дата выезда должна быть после даты заезда
    CONSTRAINT chk_dates
      CHECK (check_out_date > check_in_date)
);

-- Или триггер, который может проверять что-то более сложное,
-- например, что номер не забронирован на эти даты.
```

### **2. Возможные дополнительные вопросы и ответы**

**1. Вопрос: Что происходит при попытке удалить запись из родительской таблицы, на которую есть ссылки?**  
**Ответ:** Поведение зависит от опции `ON DELETE`, указанной при создании внешнего ключа:

- `RESTRICT` (или `NO ACTION`): **Запретить** удаление. (Стандартный и самый безопасный вариант).
    
- `CASCADE`: **Удалить** все связанные записи в дочерней таблице. (Опасный, но полезный для автоматического поддержания чистоты).
    
- `SET NULL`: Установить **NULL** в соответствующем внешнем ключе дочерней таблицы. (Требует, чтобы столбец был nullable).
    
- `SET DEFAULT`: Установить **значение по умолчанию** в соответствующем внешнем ключе.
    

**2. Вопрос: В чем разница между `PRIMARY KEY` и `UNIQUE + NOT NULL`?**  
**Ответ:**

- `PRIMARY KEY`:
    
    1. Только один на таблицу.
        
    2. Неявно создает кластерный индекс (в большинстве СУБД).
        
    3. Семантически обозначает главный идентификатор сущности.
        
- `UNIQUE + NOT NULL`:
    
    1. Может быть несколько на таблицу (например, `email`, `sku`).
        
    2. Создает некластерный уникальный индекс.
        
    3. Семантически обозначает альтернативный уникальный атрибут.
        

**3. Вопрос: Где должна обеспечиваться целостность: на уровне базы данных или на уровне приложения/ETL?**  
**Ответ:** **Всегда предпочтительнее на уровне базы данных.**

- **База данных:** Гарантирует целостность _всегда_, независимо от того, откуда пришли данные (приложение 1, приложение 2, прямой SQL-запрос, ETL-процесс). Это "последний рубеж обороны".
    
- **Приложение/ETL:** Целостность может быть случайно нарушена, если обойти приложение (напрямую подключиться к БД), или если в логике приложения есть ошибка.  
    **Правило для инженера данных:** ETL-процесс _должен_ быть готов обрабатывать ошибки целостности БД (отлавливать их и отправлять проблемные данные в стейдж для обработки), но не должен их игнорировать.

### **3. Особенности и "подводные камни"**

1. **Производительность против целостности.**
    
    - **Подводный камень:** Добавление ограничений (особенно `FOREIGN KEY`) замедляет операции `INSERT`, `UPDATE`, `DELETE`, так как СУБД必須 выполнять проверки.
        
    - **Решение/Компромисс:**
        
        - В **OLTP-системах** (онлайн-транзакции) ограничения целостности _обязательны_.
            
        - В **OLAP-системах** (хранилища данных) иногда _осознанно отказываются_ от `FOREIGN KEY` для ускорения загрузки больших объемов данных (особенно в колоночных СУБД типа ClickHouse). Целостность при этом обеспечивается ETL-процессом, который готовит данные.
            
2. **Отложенная проверка ограничений (DEFERRABLE).**
    
    - **Подводный камень:** В рамках одной транзакции может потребоваться вставить данные в родительскую и дочернюю таблицу в "неправильном" порядке, что вызовет ошибку внешнего ключа.
        
    - **Решение:** В некоторых СУБД (например, PostgreSQL) можно объявить ограничение как `DEFERRABLE`. Это позволяет _отложить проверку_ ограничения до момента commit транзакции.
```sql
-- Пример в PostgreSQL
BEGIN TRANSACTION;
SET CONSTRAINTS fk_sales_product DEFERRED;
-- Сначала вставляем в fact_sales (что без DEFERRED вызвало бы ошибку)
INSERT INTO fact_sales (sale_id, product_id) VALUES (100, 999);
-- ...а потом вставляем недостающую запись в dim_product
INSERT INTO dim_product (product_id, product_name) VALUES (999, 'New Product');
COMMIT; -- Проверка ограничения fk_sales_product произойдет здесь
```

3. **Целостность в распределенных системах.**
    
    - **Подводный камень:** В системах с репликацией или шардированием обеспечить строгую целостность в реальном времени очень сложно и дорого с точки зрения производительности.
        
    - **Решение:** Часто используется ослабленная консистентность (eventual consistency), когда данные на всех репликах становятся согласованными не мгновенно, а через некоторое время. Это компромисс, на который идут ради доступности и масштабируемости.
        

**Итог для инженера данных:** Ваша задача — понимать эти ограничения и проектировать ETL-процессы так, чтобы они либо полностью соответствовали им, либо явно и обоснованно их обходили (например, отключая FK перед загрузкой в DWH и включая после), имея план по обработке потенциально "грязных" данных.
