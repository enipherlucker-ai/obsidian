## 1. Определение

**`git cherry-pick`** — это команда, которая позволяет **взять один конкретный коммит из одной ветки и перенести его в другую**, не затрагивая остальные изменения.

- В отличие от `merge` или `rebase`, cherry-pick не объединяет всю историю, а копирует **отдельный коммит**.
- В результате в целевой ветке создаётся новый коммит с тем же изменением, но с новой ссылкой (hash).

---

## 2. Когда применять

### Типовые сценарии:

1. **Быстрый фикс (hotfix) в продакшне**
    - Ты сделал исправление бага в `develop`.
    - Нужно срочно применить то же исправление в `main`, не дожидаясь полного слияния develop.
    - Решение: `git checkout main && git cherry-pick <commit_hash>`.

2. **Выборочное применение изменений**
    - В `feature`-ветке есть несколько коммитов, но в `main` нужен только один из них.
    - Вместо мёрджа всей ветки можно сделать cherry-pick только нужного коммита.

3. **Прототипирование / перенос части работы**
    - Разработчик сделал полезное изменение (например, новый SQL для инкрементальной загрузки).
    - Можно забрать его коммит в свою ветку без полного объединения кода.

---

## 3. Примеры использования

```bash
# Найти нужный коммит (в логе видно hash)
git log --oneline

# Перейти в ветку, куда хотим перенести
git checkout main

# Перенести конкретный коммит
git cherry-pick a1b2c3d

```

**Пример для дата-инженера:**

- В `develop` кто-то добавил DAG для проверки качества данных.
- Ты хочешь протестировать его в своей `feature`-ветке.
- Делаешь cherry-pick нужного коммита → получаешь DAG, не таща весь `develop`.

---

## 4. Подводные камни

- **Конфликты**: cherry-pick может вызвать те же конфликты, что и merge. Их нужно вручную решать.
- **Дубликаты коммитов**: если потом делать merge между ветками, может появиться один и тот же коммит дважды (разные hash). Это усложняет историю.
- **История становится «грязной»**: частое использование cherry-pick затрудняет чтение логов, так как изменения «размножаются» по веткам.
- **Лучше использовать для изолированных случаев** (hotfix, выборочные правки), а не как основной инструмент.

---

## 5. Возможные вопросы на аттестации

**Вопрос:** Чем cherry-pick отличается от merge?  
**Ответ:**

- `merge` переносит всю историю изменений между ветками,
- `cherry-pick` переносит только один (или несколько) выбранных коммитов.

---

**Вопрос:** Что делать, если при cherry-pick возник конфликт?  
**Ответ:**

- Git останавливает процесс, нужно вручную разрешить конфликт,
- затем выполнить `git add .` и `git cherry-pick --continue`.

---

**Вопрос:** Можно ли cherry-pick несколько коммитов сразу?  
**Ответ:** Да. Например:
```bash
git cherry-pick a1b2c3d e4f5g6h
```
или диапазон:
```bash
git cherry-pick a1b2c3d..e4f5g6h
```
**Вопрос:** Когда cherry-pick лучше не использовать?  
**Ответ:** Когда нужно перенести большую часть изменений или целую ветку — в этом случае лучше делать merge или rebase.

---

✅ Итог:

- **Cherry-pick** — инструмент для выборочного переноса коммитов между ветками.
- Основные сценарии: hotfix, выборочное применение изменений, перенос полезного кода.
- Подводные камни: дублирование коммитов, конфликты, «засорение» истории.
- В продакшн-процессах применяется редко и точечно.
