## 1. Основные стратегии ведения репозитория

В современной разработке используют несколько популярных моделей работы с git-репозиторием:

### 1.1. **Git Flow**
- Основные ветки: `main` (или `master`) и `develop`.
- Для задач создаются отдельные `feature`-ветки от `develop`.
- После завершения разработки они вливаются в `develop`.
- Для релизов создаются `release`-ветки, которые потом мёрджатся в `main`.
- Для критических багов используются `hotfix`-ветки.

**Применение**: удобно для больших проектов с чёткой фазой релизов.  
**Минус**: избыточная сложность, много веток, тяжелее сопровождать.

---

### 1.2. **GitHub Flow**

- Есть одна основная ветка `main`.
- Каждая задача — отдельная ветка (`feature/task-123`).
- После завершения делается pull request в `main`.
- Код обязательно проходит review и тесты перед мёрджем.

**Применение**: подходит для CI/CD, где `main` всегда в рабочем состоянии.  
**Плюс**: простота, минимум веток.

---

### 1.3. **Trunk Based Development**

- Есть одна «труба» — `main` (или `trunk`).
- Разработчики часто (каждый день) мёрджат изменения в `main`.
- Минимум долгоживущих веток, задачи дробятся на маленькие коммиты.
- Тестирование и фичи прячутся за **feature flags**.

**Применение**: высокоскоростная разработка, активное CI/CD.  
**Минус**: требует дисциплины и хорошей автоматизации.

---

## 2. Какая стратегия используется в отделе

У вас применяется **гибрид между GitHub Flow и “sandbox”-веткой**:
- Есть основная ветка `main` — эталон кода, от неё все делают свои `feature`-ветки.
- После доработки задача вливается обратно в `main`.
- Есть дополнительная ветка `dev` — это **“свалка”**:
    - В ней не соблюдается чистота коммитов.
    - Все разработчики имеют доступ.
    - Используется для тестирования ETL-процессов в Airflow, быстрых экспериментов.

📌 По сути — продакшн-стратегия строится вокруг **GitHub Flow**, а `dev` выполняет роль **неформальной тестовой среды**.

---

## 3. Подводные камни

- **Force push**: если кто-то сделает `git push --force` в `main` или `dev`, можно потерять чужие изменения. Поэтому force push стоит запрещать в защищённых ветках.
- **Dev-ветка**:
    - Хорошо подходит для быстрых тестов, но со временем может превратиться в "болото".
    - Важно, чтобы в продакшн (main) попадал только чистый код.
- **Code Review**: в вашей схеме ревью обязательно для `main`, но не для `dev`. Это нормальная практика: быстрое тестирование в dev, качество — в main.
- **CI/CD**: желательно настроить пайплайны так, чтобы деплой шёл только из `main`, а из `dev` — только тестовые прогоны.

## 4. Возможные вопросы на аттестации

**Вопрос:** Почему вы выбрали именно эту стратегию?  
**Ответ:** Она сочетает простоту GitHub Flow с возможностью быстрой проверки идей в dev. Это ускоряет разработку ETL-процессов и снижает время обратной связи.

---

**Вопрос:** Чем плоха dev-ветка как свалка?  
**Ответ:** Если в ней хранить всё подряд, можно потерять воспроизводимость тестов. Хорошая практика — периодически чистить dev или сбрасывать её на состояние main.

---

**Вопрос:** Что лучше — Git Flow или GitHub Flow для дата-инженерии?  
**Ответ:** GitHub Flow проще и быстрее. В ETL-процессах важна скорость обратной связи и стабильность в main, поэтому GitHub Flow чаще применяют. Git Flow избыточен.

---

**Вопрос:** Как вы тестируете код, если dev не всегда стабильный?  
**Ответ:** Для продакшн-ETL тесты проходят только через main, а dev используется как "песочница". Таким образом, нестабильность dev не мешает работе в продакшне.

---

✅ Итог:

- Основные стратегии — Git Flow, GitHub Flow, Trunk Based Development.
- В отделе используется **GitHub Flow + dev-ветка для быстрых тестов**.
- Подводные камни: dev может превратиться в болото, force push опасен, CI/CD должен быть настроен правильно.
