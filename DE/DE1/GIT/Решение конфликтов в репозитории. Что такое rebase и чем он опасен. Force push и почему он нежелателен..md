**Конфликт** — это ситуация, когда Git не может автоматически объединить изменения из разных веток, потому что они затрагивают одни и те же участки кода/файлов.

---

#### **Типичные примеры из ETL/Airflow практики:**

**1. Конфликт в DAG-файле:**

```python
# Ветка main: другой инженер изменил расписание
with DAG(
    dag_id='sales_etl',
    schedule_interval='@daily'  # ← Кто-то поменял на daily
):
    ...

# Ваша ветка feature: вы добавляете теги
with DAG(
    dag_id='sales_etl', 
    schedule_interval='@hourly',  # ← Конфликт! Оба меняли schedule_interval
    tags=['sales', 'core']        # ← Ваше новое изменение
):
    ...
```
**2. Конфликт в SQL-скрипте витрины:**

```sql
-- Коллега в main добавил фильтр:
WHERE status = 'active'
  AND department = 'sales'  -- ← Его изменение

-- Вы в feature-ветке тоже добавили условие:  
WHERE status = 'active'
  AND is_test = false       -- ← Ваше изменение → Конфликт!
```
**3. Конфликт зависимостей:**

```txt
# requirements.txt - вы обновили pandas для новых фич
pandas==2.0.0              # ← Ваше изменение

# Коллега добавил библиотеку, которая требует старую версию  
pandas==1.5.3              # ← Его изменение → Конфликт версий!
special-library==1.0.0     # (требует pandas<2.0)
```
**4. Конфликт схемы данных:**

- Вы добавляете поле `customer_tier` в таблицу `users`
- Параллельно другой инженер добавляет поле `loyalty_score` в ту же таблицу
- Миграции конфликтуют при применении к БД.
**Основной принцип:** Предотвращать конфликты через частую интеграцию, а решать — через коммуникацию и правильные инструменты.

---

#### **1. Типичные конфликты в ETL/Airflow проектах:**

- **DAG-файлы:** Два инженера меняют один и тот же DAG (`sales_etl.py`) — один добавляет новые таски, второй меняет расписание
- **Конфиги:** Конфликт в `docker-compose.yml` или `requirements.txt` при обновлении версий пакетов
- **SQL-скрипты:** Изменения в витринах данных, где несколько разработчиков правят один и тот же CTE
- **Schema миграции:** Два инженера независимо добавляют столбцы в одну таблицу через разные миграционные файлы

#### **2. Процесс решения на практике:**

```bash

# Ежедневная практика для профилактики
git fetch origin
git rebase origin/main  # Решаем маленькие конфликты сразу

# При конфликте в MR:
1. Локально: git rebase origin/main
2. Решаю конфликты в PyCharm/VSCode с тестированием DAG
3. Запускаю тесты: pytest tests/ && airflow dags test my_etl_dag
4. Пушим: git push --force-with-lease
```
#### **3. Особенности для Airflow проектов:**

- **Перед мержем** обязательно тестирую DAG через `airflow dags test`
- **При конфликте в DAG** проверяю, не сломает ли мое решение зависимости между тасками
- **Для миграций БД** использую Liquibase/Flyway — конфликты решаются через порядок выполнения

#### **4. Критичные правила:**

- ✅ **Никогда не rebase-ить запушенные в общий репозиторий коммиты**
- ✅ **После решения конфликта запускать полный пайплайн CI/CD**
- ✅ **Сначала мержить PR с изменениями схемы, потом — с бизнес-логикой**    
- ✅ **Использовать `--force-with-lease` вместо `--force`**

**Rebase** — это операция в Git, которая "переписывает историю": переносит коммиты на новую базу (например, из `main` в свою ветку).

### В ETL/airflow работе:

- Часто мы ведём разработку DAG-ов в отдельных ветках.
- При `git rebase main` ветка как бы "накладывается" поверх свежего `main`.

### Опасность:

- История коммитов меняется → если ветку уже пушили в общий репозиторий, коллеги получат конфликты при `pull`.
- Можно потерять коммиты или случайно "раздавить" чужие изменения.
### Пример:
- Был DAG `load_sales.py` в ветке `feature/sales`.
- Коллега в `main` поменял `default_args` для всех DAG.
- Я делаю `rebase` → мои коммиты переписываются поверх новых.
- Если я уже пушил ветку, после rebase придётся делать `push --force`, что может затереть чужую работу.

### 🔨 Что такое force push

`git push --force` — это команда, которая **переписывает историю в удалённой ветке** на ту, что у тебя локально. Git не пытается объединять, а тупо заменяет.

---

### ⚠️ Почему нежелателен

1. **Затирание чужих коммитов**
    - Если кто-то уже запушил свои изменения, а ты сделал `--force`, его коммиты исчезнут из remote.
        
2. **Разъезд истории**
    - Коллеги, которые успели забрать старую ветку, столкнутся с конфликтами при `git pull`.

3. **Сложность восстановления**
    - Потерянные коммиты можно вернуть через `reflog`, но это требует лишнего времени и координации.

4. **Риск на проде**
    - Если ветка с DAG (Airflow) задеплоена из Git, force push может "откатить" код на предыдущую версию и сломать расписания или пайплайны.
---

### 👨‍💻 Пример у data engineer

- Ты ребейзнул ветку с DAG `etl_sales.py` и сделал `push --force`.
- Вчера коллега добавил в тот же DAG `sla_miss_callback`.
- После твоего пуша его кода в репозитории нет, и Airflow теряет обработку SLA.

---

### ✅ Альтернатива

- Для общих веток: использовать `merge` или `git pull --rebase` только локально.
    
- Если очень нужно перезаписать: `git push --force-with-lease` → он проверяет, не было ли новых коммитов на сервере и не сотрёт ли чужое.
