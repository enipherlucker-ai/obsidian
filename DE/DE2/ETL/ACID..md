### 18.1. Определение

**ACID** — набор свойств транзакций в реляционных СУБД, гарантирующий надёжную обработку данных:

- **A** — Atomicity (Атомарность)
- **C** — Consistency (Согласованность)
- **I** — Isolation (Изолированность)
- **D** — Durability (Долговечность)

### 18.2. Atomicity (Атомарность)

**Определение:** транзакция выполняется полностью или не выполняется совсем. Промежуточных состояний быть не может — «всё или ничего».

```sql
BEGIN;
    UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
    UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT;  -- Либо обе операции применяются, либо ни одна
```

Если между двумя UPDATE произойдёт сбой (крэш сервера, ошибка в запросе), СУБД выполнит **ROLLBACK** — откатит все изменения транзакции.

**Механизмы реализации:**
- **Undo Log (Журнал отката):** перед изменением данных СУБД записывает «старые» значения в undo-лог. При ROLLBACK «старые» значения восстанавливаются.
- **Redo Log (WAL):** записывает «новые» значения до того, как они попадут на диск. При крэше после COMMIT — данные восстанавливаются из WAL.

**Savepoints (точки сохранения):**

```sql
BEGIN;
    INSERT INTO orders VALUES (1, 'ProductA', 100);
    SAVEPOINT sp1;
    INSERT INTO orders VALUES (2, 'ProductB', 200);
    -- Ошибка! Откатываем только до savepoint
    ROLLBACK TO SAVEPOINT sp1;
    -- Первый INSERT остаётся
    INSERT INTO orders VALUES (3, 'ProductC', 300);
COMMIT;
-- В итоге вставлены записи 1 и 3
```

### 18.3. Consistency (Согласованность)

**Определение:** транзакция переводит базу данных из одного корректного состояния в другое. Все ограничения целостности должны выполняться до и после транзакции.

Виды ограничений целостности:
- **PRIMARY KEY** — уникальный идентификатор строки.
- **FOREIGN KEY** — ссылочная целостность (каскадное удаление/обновление).
- **UNIQUE** — уникальность значений.
- **CHECK** — произвольные бизнес-правила.
- **NOT NULL** — запрет NULL-значений.
- **Триггеры (TRIGGER)** — пользовательские проверки при INSERT/UPDATE/DELETE.
- **Exclusion constraints** (PostgreSQL) — запрет пересекающихся диапазонов.

```sql
CREATE TABLE accounts (
    id         SERIAL PRIMARY KEY,
    balance    NUMERIC(15,2) NOT NULL CHECK (balance >= 0),
    owner_id   INT NOT NULL REFERENCES users(id) ON DELETE CASCADE
);

-- Попытка установить отрицательный баланс — СУБД откажет
UPDATE accounts SET balance = -100 WHERE id = 1;
-- ERROR: new row for relation "accounts" violates check constraint "accounts_balance_check"
```

Если транзакция нарушает любое ограничение, она откатывается целиком.

### 18.4. Isolation (Изолированность)

**Определение:** параллельные транзакции не должны влиять друг на друга. Каждая транзакция «видит» данные так, как будто она выполняется в одиночку.

#### 18.4.1. Аномалии параллельного доступа

| Аномалия                                       | Описание                                                                                                                             | Пример                                                                                                   |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| **Dirty Read** (грязное чтение)                | Транзакция T2 читает данные, изменённые T1, но T1 ещё не сделала COMMIT. Если T1 откатится — T2 прочитала «мусор».                   | T1: UPDATE balance=0; T2: SELECT balance → 0; T1: ROLLBACK; — T2 прочитала неверное значение             |
| **Non-Repeatable Read** (неповторяемое чтение) | T2 дважды читает одну строку, но между чтениями T1 изменила и закоммитила эту строку. Результаты разные.                             | T2: SELECT balance → 1000; T1: UPDATE balance=500; COMMIT; T2: SELECT balance → 500                      |
| **Phantom Read** (фантомное чтение)            | T2 дважды выполняет один запрос, но между ними T1 вставила новые строки, подходящие под условие WHERE. Появились «фантомные» строки. | T2: SELECT COUNT(*) WHERE status='new' → 5; T1: INSERT ... status='new'; COMMIT; T2: SELECT COUNT(*) → 6 |
| **Serialization Anomaly**                      | Результат параллельного выполнения транзакций невозможно получить ни при каком последовательном порядке их выполнения.               | Две транзакции одновременно делают взаимозависимые UPDATE, создавая цикл зависимостей                    |

#### 18.4.2. Уровни изоляции (SQL Standard)

| Уровень | Dirty Read | Non-Repeatable Read | Phantom Read | Serialization Anomaly |
|---|---|---|---|---|
| **READ UNCOMMITTED** | Возможно | Возможно | Возможно | Возможно |
| **READ COMMITTED** | Нет | Возможно | Возможно | Возможно |
| **REPEATABLE READ** | Нет | Нет | Возможно | Возможно |
| **SERIALIZABLE** | Нет | Нет | Нет | Нет |

```sql
-- Установка уровня изоляции в PostgreSQL
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
    -- ...
COMMIT;
```

**Особенности PostgreSQL:**
- По умолчанию используется **READ COMMITTED**.
- READ UNCOMMITTED в PostgreSQL ведёт себя как READ COMMITTED (PostgreSQL не допускает dirty read даже на самом низком уровне).
- REPEATABLE READ в PostgreSQL защищает и от phantom read (через MVCC со snapshot).
- SERIALIZABLE реализован через **SSI (Serializable Snapshot Isolation)** — отслеживание зависимостей между транзакциями.

#### 18.4.3. MVCC (Multi-Version Concurrency Control) в PostgreSQL

PostgreSQL реализует изоляцию через **MVCC** — каждая запись хранит несколько версий:

- Каждая строка имеет скрытые поля `xmin` (ID транзакции, создавшей версию) и `xmax` (ID транзакции, удалившей/обновившей версию).
- При UPDATE создаётся новая версия строки (старая помечается `xmax`).
- Каждая транзакция «видит» только те версии строк, которые были закоммичены до начала её snapshot.
- Читатели не блокируют писателей, писатели не блокируют читателей.

```
Таблица accounts (физически):
┌────┬─────────┬─────┬──────┐
│ id │ balance │xmin │ xmax │
├────┼─────────┼─────┼──────┤
│  1 │   1000  │ 100 │  105 │  ← старая версия (удалена транзакцией 105)
│  1 │    500  │ 105 │   -  │  ← новая версия (создана транзакцией 105)
└────┴─────────┴─────┴──────┘

Транзакция 103 (начата до 105) → видит balance=1000
Транзакция 107 (начата после COMMIT 105) → видит balance=500
```

**VACUUM** — фоновый процесс, удаляющий «мёртвые» версии строк, которые больше не видны ни одной транзакции.

### 18.5. Durability (Долговечность)

**Определение:** после COMMIT данные сохранены навсегда, даже при сбое питания, крэше ОС или аппаратном сбое.

**Механизмы реализации:**

1. **WAL (Write-Ahead Log / журнал предзаписи):**
   - Прежде чем изменить данные на диске, СУБД записывает описание изменения в WAL-файл.
   - WAL записывается последовательно (sequential write) — это быстро.
   - При крэше СУБД «проигрывает» WAL и восстанавливает все закоммиченные транзакции.

2. **fsync:**
   - Системный вызов, гарантирующий, что данные из буфера ОС записаны на физический диск.
   - PostgreSQL вызывает `fsync` при записи WAL для каждого COMMIT (настройка `synchronous_commit`).

3. **Checkpoints:**
   - Периодически СУБД «сбрасывает» все грязные страницы из буферного кеша на диск.
   - После checkpoint-а WAL-файлы до этой точки можно удалить.
   - Настройка: `checkpoint_timeout`, `max_wal_size`.

```
Процесс записи:
1. BEGIN → СУБД начинает транзакцию
2. UPDATE → изменения записываются в WAL (диск) + в shared buffers (RAM)
3. COMMIT → WAL-запись о COMMIT сбрасывается на диск (fsync)
4. → Ответ клиенту: "OK"
5. (позже) Checkpoint → грязные страницы из RAM → диск
```

### 18.6. Где применяется ACID и где ослабляется

| Характеристика | OLTP (ACID) | OLAP (ослабленный ACID) |
|---|---|---|
| **Примеры СУБД** | PostgreSQL, MySQL, Oracle, SQL Server | ClickHouse, BigQuery, Redshift |
| **Транзакции** | Полноценные, короткие | Нет или ограниченные |
| **Изоляция** | READ COMMITTED — SERIALIZABLE | Обычно нет (запросы read-only) |
| **Atomicity** | Полная | Частичная (INSERT может быть атомарен, но не UPDATE) |
| **Durability** | WAL + fsync | Реплики + eventual consistency |
| **Паттерн нагрузки** | Много мелких транзакций | Немного тяжёлых аналитических запросов |

### 18.7. BASE как альтернатива ACID

**BASE** — модель, применяемая в распределённых NoSQL-системах:

- **BA — Basically Available:** система всегда отвечает, даже если часть узлов недоступна (возможно, с устаревшими данными).
- **S — Soft State:** состояние системы может меняться со временем даже без новых записей (из-за фоновой репликации).
- **E — Eventually Consistent:** данные в конечном итоге станут согласованными на всех узлах, но не гарантируется мгновенная согласованность.

| | ACID | BASE |
|---|---|---|
| **Согласованность** | Строгая (strong consistency) | Eventual consistency |
| **Доступность** | Может жертвовать доступностью | Приоритет доступности |
| **Применение** | Банки, бухгалтерия, e-commerce | Соцсети, кэши, логи, метрики |
| **Примеры** | PostgreSQL, MySQL | Cassandra, DynamoDB, MongoDB |

### 18.8. CAP-теорема (кратко)

**CAP-теорема** (Эрик Брюер, 2000) утверждает, что распределённая система может одновременно обеспечивать только два из трёх свойств:

- **C (Consistency):** все узлы видят одинаковые данные в один момент времени.
- **A (Availability):** каждый запрос получает ответ (не обязательно актуальный).
- **P (Partition Tolerance):** система продолжает работать при разрыве сети между узлами.

Поскольку **P неизбежно** в распределённых системах, реальный выбор — между **CP** (PostgreSQL с синхронной репликацией, ZooKeeper) и **AP** (Cassandra, DynamoDB).
