### 19.1. Batch Processing (пакетная обработка)

**Batch Processing** — обработка данных накопленными порциями (batch'ами). Данные сначала накапливаются за определённый период (час, день), затем обрабатываются целиком.

**Характеристики:**
- **Высокая пропускная способность (throughput):** обработка терабайтов за один запуск.
- **Высокая задержка (latency):** результат доступен через минуты/часы после поступления данных.
- **Простота:** легко отлаживать и перезапускать (данные неизменны).
- **Идемпотентность:** повторный запуск даёт тот же результат.

**Типичные инструменты:**
- **Apache Spark (batch mode)** — распределённая обработка с DAG-планированием.
- **Apache Airflow** — оркестратор ETL-пайплайнов (DAG задач).
- **Hadoop MapReduce** — классический двухфазный подход (Map → Shuffle → Reduce).
- **dbt** — трансформации в DWH через SQL.
- **Hive** — SQL-обёртка над Hadoop.

**Пример пайплайна:**

```
06:00 — Airflow запускает DAG:
  Task 1: Extract (выгрузка данных из PostgreSQL за вчерашний день)
  Task 2: Transform (Spark-джоб: очистка, агрегация, обогащение)
  Task 3: Load (загрузка результатов в ClickHouse/DWH)
  Task 4: Уведомление в Slack
06:45 — Данные доступны аналитикам
```

### 19.2. Stream Processing (потоковая обработка)

**Stream Processing** — обработка данных по мере их поступления (event-by-event или micro-batch). Каждое событие обрабатывается с минимальной задержкой.

**Характеристики:**
- **Низкая задержка (latency):** миллисекунды — секунды.
- **Непрерывная обработка:** pipeline работает 24/7.
- **Сложность:** труднее обеспечить exactly-once, обработку опоздавших данных, откат.
- **Stateful processing:** необходимо хранить состояние (агрегаты, окна).

**Ключевые концепции:**

#### 19.2.1. Windowing (оконная обработка)

| Тип окна | Описание | Пример |
|---|---|---|
| **Tumbling Window** | Фиксированные непересекающиеся окна | 5-минутные окна: [00:00–05:00), [05:00–10:00) |
| **Sliding Window** | Фиксированного размера, но с перекрытием (slide < size) | Окно 10 мин, шаг 5 мин: [00:00–10:00), [05:00–15:00) |
| **Session Window** | Группировка событий по активности с gap timeout | Сессия пользователя закрывается после 30 мин бездействия |

```
Tumbling Window (5 мин):
|──────|──────|──────|──────|
  W1      W2     W3     W4

Sliding Window (10 мин, шаг 5 мин):
|────────────|
     |────────────|
          |────────────|
  W1       W2       W3

Session Window (gap 30 мин):
|──events──|   30мин    |──events──|
   Session 1              Session 2
```

#### 19.2.2. Watermarks и Late Data

- **Event Time** — время, когда событие произошло (в источнике).
- **Processing Time** — время, когда событие обработано системой.
- **Watermark** — «метка воды», утверждающая: «все события с event time ≤ W уже получены».
- **Late Data** — события, пришедшие после watermark. Стратегии:
  - Отбросить (drop).
  - Обработать в следующем окне.
  - Allowed lateness — дополнительный тайм-аут на ожидание опоздавших.

#### 19.2.3. Гарантии доставки

| Гарантия | Описание |
|---|---|
| **At-most-once** | Событие обработается 0 или 1 раз. Потеря данных допустима. |
| **At-least-once** | Событие обработается хотя бы 1 раз. Возможны дубликаты. |
| **Exactly-once** | Событие обработается ровно 1 раз. Самая сложная гарантия. Достигается через транзакционные коммиты, идемпотентность или checkpointing. |

**Типичные инструменты:**
- **Apache Kafka Streams** — библиотека для stream processing поверх Kafka.
- **Apache Flink** — мощный движок стримовой обработки (true event-by-event, stateful, exactly-once).
- **Spark Structured Streaming** — стриминг через micro-batching в Spark.
- **Apache Kafka + ksqlDB** — SQL-интерфейс для стримовой обработки.

### 19.3. Сравнение Batch vs Stream Processing

| Критерий | Batch Processing | Stream Processing |
|---|---|---|
| **Задержка (latency)** | Минуты — часы | Миллисекунды — секунды |
| **Пропускная способность** | Очень высокая | Ниже (per-event overhead) |
| **Сложность разработки** | Низкая | Высокая (state, ordering, failures) |
| **Гарантии обработки** | Простые (перезапуск batch) | Сложные (exactly-once, checkpointing) |
| **Подход к данным** | Bounded dataset (конечный) | Unbounded stream (бесконечный) |
| **Обработка ошибок** | Перезапуск всего batch | Checkpoint + restart с offset |
| **State management** | Обычно stateless (файлы) | Stateful (RocksDB, in-memory) |
| **Типичные задачи** | ETL, отчёты, ML-обучение | Мониторинг, fraud detection, real-time dashboards |
| **Инструменты** | Spark, Airflow, dbt, Hive | Flink, Kafka Streams, Spark Streaming |

### 19.4. Lambda-архитектура

**Lambda-архитектура** (Nathan Marz, 2011) — гибридный подход, объединяющий batch и stream processing.

```
                    ┌─────────────────────┐
                    │   Источник данных   │
                    │  (Kafka, файлы...)  │
                    └──────┬──────────────┘
                           │
              ┌────────────┴────────────┐
              │                         │
    ┌─────────▼──────────┐   ┌─────────▼──────────┐
    │   Batch Layer      │   │   Speed Layer      │
    │ (Spark, MapReduce) │   │ (Flink, Storm)     │
    │                    │   │                    │
    │ • Полная история   │   │ • Только свежие    │
    │ • Точные результаты│   │ • Приблизительные  │
    │ • Перезапускаемость│   │ • Низкая задержка  │
    └─────────┬──────────┘   └─────────┬──────────┘
              │                         │
    ┌─────────▼─────────────────────────▼──────────┐
    │              Serving Layer                    │
    │  (Druid, Cassandra, ClickHouse, PostgreSQL)  │
    │                                              │
    │  Объединяет batch views + real-time views    │
    │  Клиент запрашивает = batch_result + delta   │
    └──────────────────────────────────────────────┘
```

**Три слоя:**

1. **Batch Layer** — обрабатывает всю историю данных, создаёт «batch views» (точные, полные). Запускается периодически (каждый час/день). Результат перезаписывает предыдущий batch view.
2. **Speed Layer** — обрабатывает только свежие данные (с момента последнего batch). Создаёт «real-time views» (приблизительные, но актуальные). Работает непрерывно.
3. **Serving Layer** — объединяет batch views и real-time views для ответа на запросы. Когда batch обновляется, соответствующий real-time view отбрасывается.

**Преимущества:**
- Точные результаты (batch) + низкая задержка (speed).
- Batch layer может перевычислить всё «с нуля» — самовосстановление от ошибок.
- Отказоустойчивость: если speed layer упадёт, batch layer всё равно даст результат (с задержкой).

**Недостатки:**
- **Двойная логика (dual logic problem):** одну и ту же бизнес-логику нужно реализовать дважды — в batch и в speed layer (на разных фреймворках). Сложность поддержки.
- **Сложность архитектуры:** три слоя, несколько технологий, синхронизация.
- **Стоимость:** двойной расход ресурсов.

### 19.5. Kappa-архитектура

**Kappa-архитектура** (Jay Kreps, LinkedIn, 2014) — упрощение Lambda, где остаётся **только streaming pipeline**.

```
    ┌─────────────────────┐
    │   Источник данных   │
    │  (Kafka — лог)      │
    └──────┬──────────────┘
           │
    ┌──────▼──────────────┐
    │  Stream Processing  │
    │  (Flink / Kafka     │
    │   Streams)          │
    │                     │
    │  • Единая логика    │
    │  • Replay из лога   │
    └──────┬──────────────┘
           │
    ┌──────▼──────────────┐
    │   Serving Layer     │
    │  (ClickHouse, KSQL) │
    └─────────────────────┘
```

**Ключевая идея:** если нужно пересчитать результаты (из-за бага, новой логики), можно **replay'нуть** данные из лога (Kafka с длительным retention) через **новую версию** streaming pipeline:

1. Запускается Job v2 (с исправленной логикой), читая Kafka с самого начала.
2. Job v2 записывает в новую serving table.
3. Когда Job v2 догнал текущий offset — переключаем клиентов на новую таблицу.
4. Job v1 останавливается, старая таблица удаляется.

**Преимущества:**
- Единая кодовая база (одна логика вместо двух).
- Проще поддержка и отладка.
- Меньше компонентов.

**Недостатки:**
- Replay может быть долгим (дни/недели данных).
- Kafka retention не бесконечен (стоимость хранения).
- Не подходит для сценариев, где нужны сложные batch-пересчёты (ML-модели на всей истории).

**Когда возможна Kappa:**
- Данные — append-only (логи, события, метрики).
- Kafka хранит полную историю (или достаточный retention).
- Бизнес-логика может быть выражена как streaming transformation.

### 19.6. Сравнение Lambda vs Kappa

| Критерий | Lambda | Kappa |
|---|---|---|
| **Количество pipeline** | 2 (batch + stream) | 1 (stream) |
| **Кодовая база** | Двойная (dual logic) | Единая |
| **Точность** | Batch даёт точные результаты | Зависит от quality of stream processing |
| **Пересчёт истории** | Batch layer перезапускается | Replay из Kafka-лога |
| **Сложность** | Высокая (3 слоя, 2 фреймворка) | Средняя (1 pipeline) |
| **Стоимость** | Высокая (двойные ресурсы) | Ниже |
| **Когда выбирать** | Сложная аналитика, ML | Event-driven, real-time, append-only |

### 19.7. Под-вопрос: Micro-batching

**Micro-batching** — компромиссный подход, при котором потоковые данные обрабатываются маленькими пакетами (micro-batches) с интервалом в секунды.

Используется в **Spark Structured Streaming**:

```python
df = spark.readStream \
    .format("kafka") \
    .option("subscribe", "events") \
    .load()

result = df.groupBy(
    window(col("timestamp"), "5 minutes"),
    col("event_type")
).count()

result.writeStream \
    .trigger(processingTime="10 seconds")  # micro-batch каждые 10 сек
    .format("parquet") \
    .start()
```

| Аспект | True Streaming (Flink) | Micro-batching (Spark) |
|---|---|---|
| **Модель** | Event-by-event | Маленькие batch (секунды) |
| **Задержка** | Миллисекунды | Секунды |
| **Throughput** | Высокий | Очень высокий (batch-оптимизации) |
| **Exactly-once** | Нативная поддержка | Через checkpoint + WAL |
| **Экосистема** | Flink-специфичная | Spark (batch + ML + stream в одном) |
