## Python: работа с оперативной памятью, GIL, Garbage Collector

Если рассказывать эту тему на аттестации, я бы начал так:

**Python удобный, но за удобство он платит производительностью и более сложной моделью памяти.**  
Чтобы понимать, почему одни вещи в Python работают медленно, а другие быстро, нужно разобрать три слоя:

1. как Python хранит объекты в памяти;
    
2. как он управляет их жизненным циклом через reference counting и garbage collector;
    
3. почему тяжёлые вычисления обычно выносятся в C/C++/Fortran-библиотеки, где можно обойти ограничения интерпретатора и GIL.
    

Именно на этом держится практическая работа Data Engineer: если не понимать, где Python тратит память и CPU, можно построить пайплайн, который будет тормозить, раздувать RAM и плохо масштабироваться.

---

## 1. Общее понимание: как Python работает с памятью

Важно сразу уточнить: обычно, когда мы говорим про внутреннее устройство Python, мы имеем в виду **CPython**, то есть основную реализацию Python на C.

На верхнем уровне картина такая:

- код Python выполняется интерпретатором;
    
- почти всё в Python — это **объекты**;
    
- объекты обычно живут в **куче**;
    
- переменные в коде — это не сами значения, а **ссылки на объекты**.
    

Например, когда я пишу:

x = 10  
y = x

это не означает, что число `10` просто “лежит в переменной”.  
На самом деле создаётся объект, представляющий число, и `x`, `y` хранят ссылки на него.

### Что это значит practically

В Python даже простой `int` — это не “голое число в регистре”, а полноценный объект с метаданными:

- тип объекта;
    
- счётчик ссылок;
    
- само значение;
    
- служебная информация для работы интерпретатора.
    

Из-за этого Python-объекты тяжёлые по памяти по сравнению с низкоуровневым массивом из C или NumPy.

---

## 2. Что хранится в памяти “под капотом”

У любого объекта в CPython есть служебная часть. Если упрощать, объект содержит:

- информацию о типе;
    
- счётчик ссылок;
    
- данные самого объекта.
    

То есть Python хранит не только значение, но и всё, что нужно для динамической типизации и безопасной работы runtime.

### Почему это важно

Из-за этого список чисел в Python — это не сплошной блок чисел в памяти.  
**Обычный `list` хранит массив указателей на объекты**, а сами объекты-числа лежат отдельно.

То есть для списка:

arr = [1, 2, 3, 4]

в памяти примерно такая картина:

- сам список;
    
- внутри него массив ссылок;
    
- отдельно 4 объекта `int`.
    

Это плохо для производительности, потому что:

- больше накладных расходов на память;
    
- хуже локальность данных;
    
- CPU-кэш используется менее эффективно;
    
- на каждую операцию надо разыменовывать ссылки и проверять типы.
    

---

## 3. Как Python выделяет память

В CPython есть несколько уровней работы с памятью.

### Маленькие объекты

Для небольших объектов Python использует собственный аллокатор, который часто называют **pymalloc**.  
Он работает через схему вида:

- **arenas** — большие участки памяти;
    
- **pools** — куски внутри arena;
    
- **blocks** — маленькие блоки под конкретные объекты.
    

Это сделано, чтобы не дёргать системный `malloc/free` на каждую мелочь.  
То есть Python старается быстро переиспользовать память под маленькие объекты.

### Крупные объекты

Для больших объектов память может выделяться уже через системный аллокатор.

### Практический вывод

Python часто **не спешит возвращать память обратно ОС**, даже если объект уже удалён.  
Он может оставить её у себя для повторного использования. Поэтому бывает ситуация: объект удалили, а процесс по RAM почти не уменьшился.

Для Data Engineer это важно, потому что в долгоживущих ETL-процессах или воркерах можно увидеть рост памяти, который не всегда означает классическую утечку. Иногда это особенности allocator’а и фрагментации.

## 4. Как Python удаляет объекты: reference counting

Основной механизм управления памятью в CPython — это **подсчёт ссылок**.

У каждого объекта есть **refcount** — число активных ссылок на него.  
Когда ссылка появляется — счётчик увеличивается.  
Когда ссылка исчезает — уменьшается.  
Когда счётчик становится равен нулю, объект можно уничтожить сразу.

Простой пример:

x = [1, 2, 3]  
y = x  
del x  
del y

Пока есть `y`, объект списка жив.  
Когда удаляется последняя ссылка, объект освобождается.

### Плюсы подхода

- объект часто удаляется сразу;
    
- предсказуемое поведение;
    
- просто реализовать.
    

### Минус

Reference counting **не умеет сам разрывать циклические ссылки**.

Например:

a = []  
b = []  
a.append(b)  
b.append(a)

Если внешние ссылки убрать, то `a` и `b` всё равно ссылаются друг на друга.  
Их счётчики ссылок не станут нулём, хотя фактически эти объекты уже никому не нужны.

Именно поэтому одного refcount недостаточно.

## 5. Garbage Collector: зачем он нужен

Здесь подключается **garbage collector**.

В CPython garbage collector нужен не для всего подряд, а в первую очередь для **поиска циклических ссылок**.

То есть логика такая:

- обычные объекты чаще всего удаляются через refcount;
    
- циклы добирает GC.
    

### Как он работает по сути

GC периодически проверяет объекты-контейнеры, которые могут участвовать в циклах:

- list
    
- dict
    
- set
    
- пользовательские объекты
    
- и другие контейнеры
    

И пытается понять, есть ли группы объектов, которые ссылаются друг на друга, но уже недостижимы извне.

Если такие группы найдены, их можно очистить.

### Почему это важно practically

Если разработчик не понимает разницу между refcount и GC, он может думать:  
“Я удалил переменную, почему память не ушла?”

Причины могут быть разные:

- есть ещё ссылки;
    
- есть циклические ссылки;
    
- allocator не отдал память ОС;
    
- память удерживается библиотекой или буфером.
    

Для Data Engineer это особенно заметно при обработке больших DataFrame, JSON, списков словарей и графов объектов.

## 6. GIL: что это такое и зачем он нужен

**GIL — Global Interpreter Lock**.  
Это глобальная блокировка интерпретатора в CPython, из-за которой **в один момент времени только один поток может выполнять Python bytecode** внутри одного процесса.

То есть потоков может быть несколько, но Python-код они не исполняют одновременно в полном параллелизме на нескольких ядрах.

### Зачем GIL вообще появился

Главная причина — упростить внутреннюю модель памяти и сделать работу с объектами CPython безопаснее.

Например, у объектов есть счётчик ссылок. Если несколько потоков одновременно будут менять refcount одного и того же объекта, без общей синхронизации это приведёт к race condition и повреждению памяти.

GIL сильно упрощает жизнь runtime:

- меньше сложных блокировок внутри интерпретатора;
    
- проще поддерживать объектную модель;
    
- исторически проще обеспечить корректность.
    

### Что важно сказать на собеседовании

GIL **не означает**, что Python “вообще не умеет в многопоточность”.  
Он означает более узкую вещь:

- **CPU-bound Python-код** в потоках обычно не ускоряется;
    
- **I/O-bound задачи** в потоках могут работать хорошо, потому что во время ожидания сети или диска GIL отпускается;
    
- **C-расширения** могут временно освобождать GIL и выполнять тяжёлую работу параллельно вне интерпретатора.

## 7. Почему обычные циклы Python медленные

Вот здесь начинается ключевая часть про память, производительность и библиотеки.

Допустим, у нас есть обычный Python-цикл:

result = []  
for x in data:  
    result.append(x * 2)

Почему это медленно?

Потому что на каждой итерации Python делает много работы:

1. берёт очередной объект;
    
2. интерпретирует bytecode;
    
3. проверяет типы;
    
4. извлекает значение;
    
5. создаёт новый Python-объект под результат;
    
6. обновляет ссылки и счётчики;
    
7. вызывает метод `append`.
    

То есть цикл выполняет не просто арифметику, а огромное количество служебных действий вокруг неё.

### Главное

В Python медленна не только математика сама по себе.  
Медленна именно **поэлементная работа на уровне Python-объектов**.

## 8. Почему массивы и векторизация работают быстрее

Теперь сравним это с NumPy-массивом.

В `NumPy` массив `ndarray` хранит данные иначе:

- есть один Python-объект-массив;
    
- внутри него метаданные: shape, dtype, strides;
    
- а сами данные лежат в **сыром непрерывном буфере памяти**.
    

Например, массив из `int32` может быть просто куском памяти:

[4 байта][4 байта][4 байта][4 байта]...

без отдельных Python-объектов на каждый элемент.

### Отсюда и выигрыш

Когда мы делаем:

arr * 2

операция часто выполняется не как миллион Python-итераций, а как **один вызов низкоуровневой функции**, которая проходит по буферу в C.

То есть:

- нет интерпретации Python-кода на каждый элемент;
    
- нет постоянных проверок типа для каждого числа;
    
- данные однородные;
    
- память компактная;
    
- CPU-кэш работает лучше.
    

Именно это и называют **векторизацией**: когда операция применяется сразу к большому блоку данных, а не вручную по элементам в Python-цикле.

## 9. Почему непрерывное хранение данных в памяти так важно

Это очень сильный момент для ответа “под капотом”.

### Обычный Python list

Список — это массив ссылок на объекты.  
Элементы могут лежать в памяти где угодно.

Это плохо для **cache locality**: процессору сложнее предсказывать и подгружать данные.

### NumPy ndarray

Данные лежат подряд.  
Процессор любит последовательный доступ:

- лучше работает prefetch;
    
- меньше cache miss;
    
- выше пропускная способность памяти.
    

То есть даже если бы не было GIL и интерпретатора, **непрерывный буфер сам по себе уже ускоряет вычисления**.

## 10. Роль C / C++ / Fortran в Python-библиотеках

Многие производительные библиотеки Python написаны не только и не столько на чистом Python.

Обычно схема такая:

- Python даёт удобный API;
    
- тяжёлая часть реализована на **C / C++ / Rust / Fortran**;
    
- Python-код только вызывает нативную функцию и получает результат.
    

### Типичные примеры

- **NumPy** — много низкоуровневого кода на C;
    
- **SciPy** — активно использует C/C++ и исторически Fortran;
    
- **pandas** — критичные части реализованы в C/Cython, а в новых сценариях может опираться и на Arrow;
    
- **Polars** — ядро на Rust;
    
- **PyArrow** — columnar memory format и операции на C++.
    

### Почему тут вообще Fortran

Fortran исторически очень силён в численных вычислениях и линейной алгебре.  
Многие зрелые и очень быстрые математические библиотеки написаны именно на нём.

Для Python это не проблема: Python может вызывать код, написанный на других языках, через расширения и биндинги.

## 11. SIMD: что это и почему ускоряет вычисления

**SIMD** — это _Single Instruction, Multiple Data_.  
Идея в том, что одна инструкция CPU может обработать сразу несколько значений.

Например, не 1 число + 1 число, а сразу 4, 8 или 16 чисел — в зависимости от архитектуры и типа данных.

Это реализуется через векторные инструкции процессора, например семейства SSE, AVX и другие.

### Почему это помогает

Если у нас массив чисел в непрерывной памяти и одного типа, то библиотека может пройти по нему векторными инструкциями.

Это ещё одна причина, почему `NumPy` и похожие библиотеки сильно быстрее Python-циклов:

- они работают не с “объектами Python”,
    
- а с плотным числовым буфером,
    
- который хорошо ложится на SIMD.

## 12. Что такое BLAS и LAPACK

Это уже уместно, если разговор заходит в численные библиотеки.

### BLAS

**BLAS** — базовые операции линейной алгебры:

- скалярные и векторные операции;
    
- матрично-векторные операции;
    
- матрично-матричное умножение.
    

### LAPACK

**LAPACK** — более сложные алгоритмы линейной алгебры:

- разложения;
    
- решение систем;
    
- собственные значения;
    
- инверсии и так далее.
    

### Почему это важно в Python

Когда мы вызываем:

A @ B

или функции из `numpy.linalg` / `scipy.linalg`, под капотом Python часто доходит до высокооптимизированных BLAS/LAPACK-реализаций:

- OpenBLAS
    
- Intel MKL
    
- ATLAS и другие
    

Это уже серьёзный нативный код, который умеет:

- эффективно работать с памятью;
    
- использовать SIMD;
    
- использовать несколько потоков;
    
- обходить Python-интерпретатор в горячем участке.

## 13. Как векторизация помогает обходить часть ограничений Python и GIL

Это очень важная связка.

Когда мы говорим, что Python медленный из-за GIL, это не совсем полная картина.  
Проблема не только в GIL, а ещё и в том, что чистый Python делает работу поэлементно через интерпретатор.

### Что делает векторизация

Она выносит горячий цикл из Python в нативный код.

То есть вместо:

for x in data:  
    ...

мы делаем:

arr * 2

и дальше цикл выполняется уже в C/C++/Fortran-коде библиотеки.

### Почему это частично обходит ограничения Python

Потому что:

1. Python не интерпретирует цикл по каждому элементу;
    
2. библиотека работает напрямую с буфером памяти;
    
3. в ряде случаев нативный код может **освободить GIL** на время вычислений;
    
4. сама нативная библиотека может использовать многопоточность и SIMD.
    

То есть формально Python всё ещё существует, но “тяжёлая работа” идёт уже не внутри медленного Python-цикла.

## 14. Сравнение: обычные циклы Python vs векторизованные операции

### Обычные циклы Python

Подход:

result = []  
for x in data:  
    result.append(x * 2)

Характеристики:

- каждый элемент — Python-объект;
    
- цикл исполняется интерпретатором;
    
- много проверок и накладных расходов;
    
- плохо масштабируется на большие объёмы;
    
- CPU-bound код страдает от GIL.
    

### Векторизованные операции

Подход:

import numpy as np  
  
arr = np.array(data, dtype=np.int64)  
result = arr * 2

Характеристики:

- данные хранятся компактно;
    
- цикл скрыт внутри нативной реализации;
    
- меньше накладных расходов;
    
- лучше кэш-локальность;
    
- можно использовать SIMD;
    
- часть операций может выполняться с освобождением GIL;
    
- зачастую намного быстрее на больших массивах.
    

---

## 15. Короткий пример сравнения

### Вариант 1. Обычный Python

data = list(range(1_000_000))  
result = [x * 2 for x in data]

Здесь миллион раз создаются и обрабатываются Python-объекты.

### Вариант 2. NumPy

import numpy as np  
  
arr = np.arange(1_000_000, dtype=np.int64)  
result = arr * 2

Здесь создаётся плотный числовой массив, и операция делается в нативном коде по непрерывному буферу.

### Как это объяснить вслух

Во втором случае Python не крутит миллион итераций сам. Он один раз вызывает быструю функцию библиотеки, которая уже эффективно проходит по памяти.

## 16. Как библиотеки работают с памятью на практике

Вот хороший ответ для аттестации:

**Производительные библиотеки стараются минимизировать количество Python-объектов и работать через сырые буферы памяти.**

Например:

### NumPy

- хранит однородные данные в `ndarray`;
    
- использует `dtype`, `shape`, `strides`;
    
- умеет делать представления без копирования, если меняется только “вид” на те же данные.
    

### pandas

- даёт табличную модель поверх более эффективных внутренних структур;
    
- векторизованные столбцовые операции обычно быстрее построчных циклов;
    
- но надо помнить, что `object`-dtype может снова вернуть нас ближе к миру Python-объектов и ухудшить производительность.
    

### PyArrow

- columnar-хранение;
    
- удобен для обмена данными между системами;
    
- хорош для аналитических и data engineering-сценариев;
    
- помогает уменьшать накладные расходы на сериализацию и копирование.
    

### Polars

- работает по колонкам;
    
- ядро не на Python;
    
- хорошо показывает, как “тонкий Python API + быстрое нативное ядро” даёт хороший результат.

## 17. Почему это важно именно для Data Engineer

Для Data Engineer эта тема не теоретическая, а очень практическая.

### 1. Объём данных

Если обрабатывать большие объёмы построчно через Python-циклы, всё быстро упрётся в:

- CPU;
    
- RAM;
    
- время выполнения DAG/джобы.
    

### 2. Выбор правильного инструмента

Нужно понимать, когда использовать:

- чистый Python;
    
- `numpy/pandas`;
    
- `polars/pyarrow`;
    
- Spark/Dask/Ray;
    
- multiprocessing вместо threading.
    

### 3. Контроль памяти

Если не понимать, как живут объекты и когда они реально удаляются, можно получить:

- высокий расход памяти;
    
- долгую работу GC;
    
- нестабильные воркеры;
    
- OOM в ETL-процессах.
    

### 4. Производительность пайплайна

Очень часто ускорение достигается не “добавлением серверов”, а тем, что:

- убирают Python-циклы;
    
- уменьшают число объектов;
    
- используют векторизацию;
    
- переходят на columnar-представление;
    
- выносят горячие участки в нативные библиотеки.

## 19. Как это лучше закончить на аттестации

Я бы завершил так:

**Итог такой:**  
Python сам по себе удобен, но его объектная модель тяжёлая: каждый объект содержит метаданные, живёт в куче и управляется через reference counting. Garbage collector нужен для циклических ссылок. GIL упрощает безопасную работу интерпретатора, но ограничивает параллельное выполнение Python bytecode в потоках.

Поэтому для тяжёлых вычислений Python обычно использует другой путь: данные кладутся в компактные структуры памяти, а горячие операции выполняются в нативном коде на C/C++/Fortran. За счёт этого мы получаем непрерывное хранение данных, лучшую работу CPU-кэша, SIMD, а иногда и многопоточность вне GIL.  
Именно поэтому векторизованные операции и специализированные библиотеки так важны для Data Engineer: они позволяют обрабатывать большие объёмы данных быстрее, стабильнее и с меньшим расходом ресурсов.

# Краткий итог

- В CPython почти всё — объекты в куче.
    
- Переменные хранят ссылки на объекты.
    
- Основной механизм удаления объектов — **reference counting**.
    
- **Garbage collector** нужен, чтобы убирать циклические ссылки.
    
- **GIL** не даёт нескольким потокам одновременно исполнять Python bytecode в одном процессе.
    
- Обычные циклы Python медленные, потому что работают поэлементно через интерпретатор и объекты.
    
- Векторизация быстрая, потому что данные лежат компактно, а цикл выполняется в нативной библиотеке.
    
- C/C++/Fortran-ядро, непрерывная память, SIMD, BLAS/LAPACK — это и есть основа высокой производительности численных библиотек.
    
- Для Data Engineer это важно из-за объёма данных, RAM, времени выполнения пайплайнов и выбора правильного инструмента.

# Возможные вопросы на аттестации и сильные короткие ответы

## 1. Почему Python ест много памяти?

Потому что значения в Python — это объекты с метаданными, а не просто сырые числа. У объекта есть тип, refcount и служебная часть, поэтому накладные расходы на память высокие.

## 2. Чем `list` отличается от `numpy.ndarray` по памяти?

`list` хранит ссылки на отдельные Python-объекты, а `numpy.ndarray` обычно хранит однородные данные в одном непрерывном буфере памяти. Поэтому `ndarray` компактнее и быстрее.

## 3. Что делает reference counting?

Следит, сколько ссылок у объекта. Когда ссылок становится ноль, объект можно удалить сразу.

## 4. Зачем тогда нужен garbage collector?

Потому что reference counting не умеет сам удалять циклические ссылки. GC ищет такие циклы и очищает их.

## 5. Что такое GIL?

Это глобальная блокировка интерпретатора в CPython. Она позволяет только одному потоку одновременно выполнять Python bytecode внутри процесса.

## 6. Значит, потоки в Python бесполезны?

Нет. Для I/O-bound задач потоки полезны, потому что во время ожидания сети, файлов или сокетов GIL обычно отпускается. Плохо они подходят именно для CPU-bound Python-кода.

## 7. Как тогда ускорять CPU-bound задачи?

Через multiprocessing, через векторизацию, через нативные библиотеки, либо выносить горячие участки из Python в C/C++/Rust/Fortran.

## 8. Почему NumPy быстрее цикла `for`?

Потому что NumPy работает с плотным буфером однородных данных и выполняет цикл в нативном коде, а не в интерпретаторе Python по объектам.

## 9. Что даёт непрерывное хранение данных?

Лучшую cache locality, меньше пропусков кэша, более эффективный последовательный доступ и возможность использовать SIMD.

## 10. Что такое SIMD простыми словами?

Это когда одна инструкция процессора обрабатывает сразу несколько элементов данных. Это особенно эффективно для массивов чисел одинакового типа.

## 11. Где тут BLAS и LAPACK?

Это высокооптимизированные библиотеки линейной алгебры. Python-библиотеки, например NumPy и SciPy, часто используют их для быстрых матричных операций и разложений.

## 12. Почему векторизация частично “обходит” GIL?

Потому что тяжёлая работа уходит из Python в нативный код библиотеки. Python не крутит цикл сам, а библиотека может выполнять вычисления эффективнее и иногда с освобождением GIL.

## 13. Почему `pandas.apply()` часто не так хорош, как кажется?

Потому что он нередко возвращает нас к поэлементной логике на уровне Python, а это убирает часть преимуществ настоящей векторизации.

## 14. Что важнее для Data Engineer: знать GIL или знать memory layout?

И то и другое, но с практической точки зрения memory layout и понимание, где создаются Python-объекты, часто важнее для производительности ETL и обработки больших таблиц.

## 15. Когда стоит уходить из pandas в Polars, Arrow или Spark?

Когда данные уже слишком большие, память расходуется неэффективно, а построчная обработка и Python-объекты становятся узким местом.
