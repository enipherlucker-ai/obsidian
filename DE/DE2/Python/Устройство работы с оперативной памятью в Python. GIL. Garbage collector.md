### 9.1. Объектная модель Python — «Всё есть объект»

В CPython **каждое** значение (число, строка, функция, класс, модуль) — это объект, расположенный в **куче** (heap). На уровне C каждый объект представлен структурой `PyObject`:

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;   // счётчик ссылок
    PyTypeObject *ob_type;  // указатель на тип
} PyObject;
```

Для объектов переменного размера (list, str, bytes) используется расширенная структура:

```c
typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;     // количество элементов
} PyVarObject;
```

Переменные в Python — это **имена** (ссылки), указывающие на объекты в куче. Присваивание `a = b` не копирует данные, а создаёт ещё одну ссылку на тот же объект.

### 9.2. Heap vs Stack

| Область | Что хранится | Управление |
|---------|-------------|------------|
| **Stack** (стек вызовов) | Фреймы функций (`PyFrameObject`), локальные ссылки (указатели на объекты) | Автоматически при входе/выходе из функции |
| **Heap** (куча) | Сами объекты Python (int, str, list, dict, экземпляры классов) | Управляется менеджером памяти Python + GC |

Стек хранит только **ссылки** (указатели); сами данные всегда живут в куче.

### 9.3. Аллокатор памяти CPython (pymalloc)

CPython использует многоуровневую систему выделения памяти:

```
Уровень 3:  Объектно-специфичные аллокаторы (int, list, dict free lists)
Уровень 2:  pymalloc — для объектов ≤ 512 байт
Уровень 1:  malloc / calloc / realloc (libc)
Уровень 0:  ОС (mmap, brk, VirtualAlloc)
```

**pymalloc** оптимизирован для частого выделения маленьких объектов:

- **Arena** (арена) — блок 256 КБ, запрашиваемый у ОС через `mmap`. Арены могут быть возвращены ОС, когда полностью свободны.
- **Pool** (пул) — 4 КБ (размер страницы). Каждый пул обслуживает блоки одного фиксированного размера (size class).
- **Block** (блок) — от 8 до 512 байт с шагом 8 байт (64 класса размеров). Один запрос на выделение получает ровно один блок.

```
Arena (256 KB)
├── Pool 0  (size class = 32 bytes) ← блоки по 32 байта
│   ├── Block
│   ├── Block
│   └── ...
├── Pool 1  (size class = 64 bytes)
│   ├── Block
│   └── ...
└── ...
```

Для объектов > 512 байт pymalloc не используется — запрос идёт напрямую к `malloc`.

### 9.4. Подсчёт ссылок (Reference Counting)

Основной механизм управления памятью в CPython — **подсчёт ссылок**.

**Принцип работы:**
- Каждый объект содержит поле `ob_refcnt`.
- При создании новой ссылки на объект счётчик увеличивается (`Py_INCREF`).
- При удалении ссылки (del, выход из области видимости, переприсваивание) — уменьшается (`Py_DECREF`).
- Когда `ob_refcnt` достигает 0, объект **немедленно** уничтожается, а его память освобождается.

```python
import sys

a = [1, 2, 3]
print(sys.getrefcount(a))  # 2 (a + временная ссылка в getrefcount)

b = a
print(sys.getrefcount(a))  # 3

del b
print(sys.getrefcount(a))  # 2
```

**Достоинства:**
- **Детерминированность** — объект уничтожается сразу, как только на него нет ссылок.
- **Немедленное освобождение** — деструктор (`__del__`) вызывается предсказуемо.
- Минимальная задержка (нет пауз на сборку мусора).

**Недостатки:**
- Не обнаруживает **циклические ссылки** (A→B→A).
- Потокобезопасность требует GIL (атомарный инкремент/декремент).
- Накладные расходы на каждое присваивание.

### 9.5. Циклический сборщик мусора (Cyclic GC)

Подсчёт ссылок не способен обнаружить циклы. Для этого в CPython встроен **трассирующий сборщик мусора с поколениями**.

**Поколения (generations):**

| Поколение | Описание | Порог по умолчанию |
|-----------|----------|--------------------|
| 0 (young) | Новые объекты | 700 (аллокаций − деаллокаций) |
| 1 (middle) | Пережили 1 сборку | 10 (сборок поколения 0) |
| 2 (old) | Пережили 2+ сборки | 10 (сборок поколения 1) |

**Алгоритм:**
1. Сборка поколения N собирает **все** поколения 0..N.
2. Для каждого контейнерного объекта (list, dict, set, экземпляры классов) формируется граф ссылок.
3. Алгоритм вычитает внутренние ссылки. Объекты с остаточным счётчиком 0 — «мусор».
4. Выжившие объекты перемещаются в следующее поколение.

**Гипотеза поколений**: большинство объектов умирают молодыми, поэтому собирать поколение 0 можно часто и дёшево.

```python
import gc

# Посмотреть текущие пороги
print(gc.get_threshold())  # (700, 10, 10)

# Установить свои пороги
gc.set_threshold(1000, 15, 15)

# Принудительная полная сборка
gc.collect()  # возвращает количество собранных объектов

# Отключить автоматическую сборку (например, при batch-обработке)
gc.disable()
# ... работа ...
gc.enable()

# Найти объекты с циклическими ссылками
gc.set_debug(gc.DEBUG_SAVEALL)
gc.collect()
print(gc.garbage)  # список недостижимых объектов с финализаторами
```

**Пример циклической ссылки:**

```python
class Node:
    def __init__(self):
        self.parent = None
        self.children = []

a = Node()
b = Node()
a.children.append(b)
b.parent = a  # цикл: a → b → a

del a, b  # refcount != 0, но объекты недостижимы → GC соберёт
```

### 9.6. GIL (Global Interpreter Lock)

**GIL** — это мьютекс (блокировка) уровня интерпретатора CPython, который гарантирует, что **в каждый момент времени только один поток** выполняет байт-код Python.

**Зачем нужен GIL:**
- `ob_refcnt` — обычная переменная (не атомарная). Без GIL одновременный `Py_INCREF` / `Py_DECREF` из разных потоков вызвал бы data race.
- GIL упрощает реализацию CPython и C-расширений (не нужны fine-grained locks для каждого объекта).

**Как работает:**
- Поток захватывает GIL перед выполнением байт-кода.
- Через каждые ~5 мс (с Python 3.2+) интерпретатор проверяет, ждёт ли GIL другой поток, и при необходимости отпускает его.
- GIL **освобождается** при:
  - **I/O операциях** (чтение файла, сетевой запрос, `time.sleep`).
  - **Вызове C-расширений**, использующих макросы `Py_BEGIN_ALLOW_THREADS` / `Py_END_ALLOW_THREADS` (NumPy, OpenCV, etc.).

**Последствия:**

```python
import threading, time

def cpu_bound():
    total = 0
    for i in range(10_000_000):
        total += i

# Последовательно: ~1.2 с
# Два потока: ~1.2 с (НЕ быстрее, потому что GIL)
# Два процесса (multiprocessing): ~0.6 с ← настоящий параллелизм
```

**Способы обхода GIL:**

| Подход | Когда использовать |
|--------|--------------------|
| `multiprocessing` | CPU-bound задачи, нужен настоящий параллелизм |
| `concurrent.futures.ProcessPoolExecutor` | То же, удобнее API |
| `threading` | I/O-bound задачи (GIL отпускается на I/O) |
| `asyncio` | Тысячи I/O-операций (один поток, event loop) |
| C-расширения с `Py_BEGIN_ALLOW_THREADS` | Вычисления в C/C++ (NumPy, pandas) |
| Субинтерпретаторы (PEP 734, Python 3.12+) | Изоляция без fork |
| Свободнопоточный CPython (PEP 703, Python 3.13+) | Экспериментальный `--disable-gil` |

### 9.7. Фрагментация памяти

Python-процесс часто **не возвращает** память ОС после освобождения объектов:
- `pymalloc` переиспользует блоки внутри арен. Арена возвращается ОС только когда **все** её пулы свободны.
- При чередовании alloc/free маленьких и больших объектов арены «прореживаются», но не освобождаются полностью.
- **Решения:** пулы объектов, `__slots__`, обработка данных порциями (chunked processing), использование `gc.collect()` + `malloc_trim()`.

### 9.8. Полезные инструменты

```python
import sys

# Размер объекта в байтах (без учёта вложенных объектов)
print(sys.getsizeof(42))         # 28 байт
print(sys.getsizeof("hello"))    # 54 байт
print(sys.getsizeof([1, 2, 3]))  # 88 байт (заголовок списка, без элементов)
```

### 9.9. Оптимизация `__slots__`

По умолчанию каждый экземпляр класса хранит атрибуты в `__dict__` (словарь). `__slots__` заменяет словарь фиксированным массивом, экономя ~40–60% памяти:

```python
class PointDict:
    def __init__(self, x, y):
        self.x = x
        self.y = y

class PointSlots:
    __slots__ = ('x', 'y')
    def __init__(self, x, y):
        self.x = x
        self.y = y

import sys
d = PointDict(1, 2)
s = PointSlots(1, 2)
print(sys.getsizeof(d) + sys.getsizeof(d.__dict__))  # ~152 байт
print(sys.getsizeof(s))                                # ~56 байт
```

Ограничения: нельзя добавлять произвольные атрибуты, сложнее наследование.

### 9.10. Слабые ссылки (weak references)

Слабая ссылка (`weakref.ref`) указывает на объект, **не увеличивая** его `ob_refcnt`. Если объект удалён, слабая ссылка возвращает `None`.

```python
import weakref

class MyObj:
    pass

obj = MyObj()
weak = weakref.ref(obj)
print(weak())       # <__main__.MyObj object at 0x...>

del obj
print(weak())       # None — объект уже собран
```

**Применение:** кэши (`weakref.WeakValueDictionary`), наблюдатели, избежание циклических ссылок в древовидных структурах (ребёнок→родитель).
