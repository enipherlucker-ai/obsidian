## Введение

Если говорить просто, и `multithreading`, и `multiprocessing` — это способы выполнять несколько задач одновременно или почти одновременно.

Но между ними есть принципиальная разница:

- **multithreading** — это несколько **потоков внутри одного процесса**
- **multiprocessing** — это несколько **отдельных процессов**

На практике это влияет на всё: на память, на скорость, на взаимодействие задач, на сложность обмена данными и на то, в каких случаях это вообще имеет смысл использовать.

Для Python это особенно важно из-за **GIL** — Global Interpreter Lock, то есть глобальной блокировки интерпретатора в CPython.

---

## Сначала базово: что такое процесс и что такое поток

### Процесс

**Процесс** — это отдельный экземпляр программы в операционной системе.

У процесса есть:

- своё адресное пространство в памяти
- свои системные ресурсы
- свои дескрипторы файлов
- свой интерпретатор Python
- своё состояние выполнения

Если процессов несколько, ОС воспринимает их как отдельные независимые сущности.

### Поток

**Поток** — это более лёгкая единица выполнения внутри процесса.

Потоки одного процесса:

- работают в **общем адресном пространстве**
- видят одни и те же объекты в памяти
- разделяют ресурсы процесса
- отличаются в основном своим стеком вызовов, регистровым состоянием и текущей точкой выполнения

То есть если очень грубо:

- процесс — это отдельная “квартира”
- поток — это несколько людей внутри одной квартиры, которые пользуются общей кухней, ванной и мебелью

# Multithreading

## Что это такое

`multithreading` — это запуск нескольких потоков внутри одного процесса.

В Python это обычно делают через модуль `threading` или через `concurrent.futures.ThreadPoolExecutor`.

### Когда это полезно

Потоки особенно хороши для **I/O-bound задач**, то есть когда программа много времени:

- ждёт ответа от сети
- читает или пишет файлы
- работает с сокетами
- ждёт базу данных
- общается с внешними API

Почему? Потому что во время ожидания поток может “отпустить управление”, и другой поток в это время может делать полезную работу.

---

## Как это работает под капотом

Под капотом поток — это сущность ОС, которую планировщик может запускать на выполнение.

Но в Python есть важный нюанс: в стандартной реализации **CPython** есть **GIL**.

### Что такое GIL

GIL — это глобальная блокировка интерпретатора, которая гарантирует, что в один конкретный момент времени **только один поток исполняет Python bytecode** внутри одного процесса.

Это означает:

- потоков может быть много
- но Python-код одновременно реально исполняется только одним из них

Из-за этого многопоточность в CPython **не даёт настоящего параллелизма для CPU-bound задач**.

Например, если у меня есть две тяжёлые вычислительные функции на Python, два потока не ускорят выполнение так, как хотелось бы. Они будут по очереди получать GIL.

---

## Почему тогда потоки вообще нужны

Потому что во время операций ввода-вывода GIL часто освобождается.

Например:

- поток отправил запрос в БД и ждёт ответ
- в этот момент другой поток может выполняться

То есть для I/O-задач многопоточность отлично подходит.

---

## Что потоки хранят и что разделяют

У каждого потока есть:

- свой стек вызовов
- свой instruction pointer, то есть текущая точка выполнения
- локальные переменные функций — фактически через стек и frame’ы

Но при этом потоки разделяют:

- кучу, heap, где лежат объекты Python
- глобальные переменные
- общие структуры данных
- открытые соединения и ресурсы процесса

### Что из этого следует

Это даёт плюс:

- обмен данными между потоками дешёвый, потому что не надо сериализовать объекты и передавать их между процессами

Но даёт и минус:

- легко получить **race condition**
- можно испортить общее состояние
- нужны `Lock`, `RLock`, `Semaphore`, `Event`, `Condition`, `Queue`

---

## Пример, где потоки полезны

Например, нужно параллельно сходить в 100 HTTP API.

Если делать последовательно:

- отправили запрос
- ждём
- получили ответ
- идём дальше

Если использовать потоки:

- можно держать много запросов “в полёте”
- пока один поток ждёт сеть, другой уже работает

### Упрощённый пример
```python
from concurrent.futures import ThreadPoolExecutor  
import requests  
  
urls = [  
    "https://example.com/1",  
    "https://example.com/2",  
    "https://example.com/3",  
]  
  
def fetch(url):  
    return requests.get(url).status_code  
  
with ThreadPoolExecutor(max_workers=3) as executor:  
    results = list(executor.map(fetch, urls))  
  
print(results)
```
Здесь ускорение есть, потому что основное время уходит не в вычисления, а в ожидание сети.

# Multiprocessing

## Что это такое

`multiprocessing` — это запуск нескольких **отдельных процессов**.

В Python обычно используют:

- модуль `multiprocessing`
- `concurrent.futures.ProcessPoolExecutor`

### Когда это полезно

Multiprocessing нужен в первую очередь для **CPU-bound задач**, то есть тяжёлых вычислений:

- парсинг больших объёмов данных
- расчёты
- компрессия
- преобразования
- обработка изображений
- часть ETL/ELT-вычислений
- большие циклы на чистом Python

---

## Как это работает под капотом

Поскольку каждый процесс — это отдельный процесс ОС, у него:

- своя память
- свой интерпретатор Python
- свой GIL

И вот это ключевой момент:

### Почему multiprocessing обходит ограничение GIL

Потому что GIL существует **внутри одного процесса**.

Если у нас 4 процесса, то у нас фактически 4 независимых интерпретатора Python и 4 отдельных GIL. Значит, процессы могут реально выполняться параллельно на разных ядрах CPU.

---

## Что хранится в процессе

У каждого процесса своё:

- адресное пространство
- heap
- стек
- набор объектов Python
- глобальные переменные модуля

То есть данные между процессами **не общие по умолчанию**.

Если один процесс изменил переменную, другой этого автоматически не увидит.

---

## Как процессы обмениваются данными

Для обмена данными между процессами используют:

- `multiprocessing.Queue`
- `Pipe`
- `Manager`
- `SharedMemory`
- файлы
- сокеты
- базы данных
- брокеры сообщений

Но здесь есть важная цена: данные часто нужно **сериализовать**, обычно через `pickle`.

### Почему это важно

Если между процессами гонять большие объекты:

- растут накладные расходы на сериализацию
- растут накладные расходы на копирование памяти
- можно потерять выигрыш от параллелизма

То есть multiprocessing — это не “бесплатное ускорение”, а компромисс.

---

## Пример, где multiprocessing полезен

Например, нужно посчитать что-то тяжёлое для большого списка чисел.
``` python
from concurrent.futures import ProcessPoolExecutor  
  
def heavy(x):  
    total = 0  
    for i in range(10_000_000):  
        total += x * i  
    return total  
  
values = [1, 2, 3, 4]  
  
with ProcessPoolExecutor(max_workers=4) as executor:  
    results = list(executor.map(heavy, values))  
  
print(results)
```
Здесь multiprocessing может реально ускорить выполнение, потому что задачи вычислительные и могут идти на разных ядрах.

# Главное различие между multithreading и multiprocessing

## 1. Единица параллелизма

- **multithreading** — несколько потоков внутри одного процесса
- **multiprocessing** — несколько независимых процессов

## 2. Память

- потоки разделяют общую память процесса
- процессы имеют раздельную память

## 3. Стоимость создания

- поток создать дешевле
- процесс создать дороже

## 4. Обмен данными

- между потоками проще и быстрее, потому что память общая
- между процессами сложнее: нужны IPC-механизмы и часто сериализация

## 5. Безопасность

- потоки опаснее с точки зрения гонок и порчи общего состояния
- процессы лучше изолированы: сбой одного меньше влияет на другие

## 6. GIL и CPU-bound задачи

- потоки в CPython не дают хорошего ускорения для CPU-bound задач
- процессы дают, потому что каждый процесс имеет свой интерпретатор и свой GIL

---

# Очень важный момент: GIL

На собеседовании почти наверняка спросят именно про это.

## Как правильно сказать

В CPython многопоточность ограничена GIL. Это значит, что в одном процессе одновременно Python bytecode исполняет только один поток. Поэтому для I/O-bound задач потоки полезны, потому что во время ожидания ввода-вывода GIL освобождается. А для CPU-bound задач лучше использовать multiprocessing, потому что процессы работают независимо и могут занимать разные ядра CPU.

Это одна из самых важных мыслей всей темы.

# Как это выглядит “под капотом” чуть глубже

## Под капотом у потоков

Когда мы создаём поток:

1. ОС создаёт новый thread execution context
2. у потока появляется свой стек
3. он начинает выполнять target-функцию
4. потоки конкурируют за CPU и за GIL
5. если поток ждёт I/O, планировщик и интерпретатор могут дать исполняться другому

### Где лежат данные

- локальные переменные функции — в frame’ах и стеке вызовов конкретного потока
- объекты Python — в heap процесса, то есть общие для всех потоков

Поэтому если два потока работают с одним списком или словарём, они реально работают с одним и тем же объектом.

---

## Под капотом у процессов

Когда создаём процесс:

1. ОС создаёт новый процесс
2. у него своя память
3. запускается новый интерпретатор Python
4. нужно передать в него задачу и аргументы
5. результат нужно вернуть обратно

### Важный практический нюанс

На разных ОС запуск процессов отличается:

- в Linux часто возможен `fork`
- в Windows используется `spawn`
- в macOS тоже часто используют `spawn` как более безопасный вариант

### Почему это важно

#### fork

При `fork` дочерний процесс создаётся как копия родительского. Изначально память может использовать механизм **copy-on-write**: пока данные не меняются, физически они могут не копироваться сразу.

Это быстро, но у `fork` есть тонкости, особенно если у родителя уже были потоки, соединения, открытые ресурсы.

#### spawn

При `spawn` запускается новый чистый процесс, который заново импортирует модуль и создаёт объекты.

Это безопаснее в ряде сценариев, но медленнее.

Из-за этого multiprocessing иногда имеет заметный overhead на старте.

# Где лучше использовать multithreading

## Подходит для:

- запросов к API
- чтения файлов
- ожидания ответов от БД
- сетевого взаимодействия
- многократных операций ввода-вывода
- случаев, где важна высокая отзывчивость

## Плюсы

- лёгкие по сравнению с процессами
- дешёвое переключение
- удобный обмен данными
- хороши для I/O

## Минусы

- GIL мешает CPU-bound параллелизму
- сложнее отлаживать из-за race condition
- общий доступ к памяти создаёт риск ошибок

---

# Где лучше использовать multiprocessing

## Подходит для:

- тяжёлых вычислений
- CPU-bound обработки данных
- параллельных расчётов
- случаев, где нужно загрузить несколько ядер процессора

## Плюсы

- настоящий параллелизм
- обход GIL
- хорошая изоляция между задачами

## Минусы

- дороже запуск
- сложнее обмен данными
- нужно сериализовать объекты
- больше расход памяти

# Что важно для Data Engineer

Для Data Engineer тему лучше связывать с реальными задачами.

## Когда полезен multithreading

Например:

- параллельно читать данные из нескольких API
- качать файлы из S3 или HTTP
- отправлять несколько запросов к внешним сервисам
- работать с сетевыми операциями в ETL

Здесь bottleneck — не CPU, а ожидание.

## Когда полезен multiprocessing

Например:

- тяжёлый парсинг больших JSON или XML
- CPU-intensive трансформации
- расчёт хешей
- предобработка файлов
- параллельная обработка чанков данных на чистом Python

Здесь bottleneck — процессор.

# Важная практическая оговорка

На практике в Data Engineering часто оказывается, что ни потоки, ни процессы — не главное средство параллелизма.

Потому что часто используются:

- Spark
- Dask
- Airflow
- базы данных, которые сами умеют выполнять запросы параллельно
- векторизованные операции pandas / NumPy
- pushdown logic в SQL-движки

То есть хороший инженер не просто “включает multiprocessing”, а сначала понимает, **где узкое место**:

- CPU?
- сеть?
- диск?
- база?
- сериализация?
- межпроцессный обмен?

# Частые ошибки и misconceptions

## 1. “Потоки всегда ускоряют программу”

Нет. Для CPU-bound задач в CPython чаще всего не ускоряют из-за GIL.

## 2. “Процессы всегда лучше”

Нет. У них дороже запуск, выше расход памяти и дороже обмен данными.

## 3. “Если много ядер, надо всегда multiprocessing”

Не всегда. Если задача I/O-bound, потоки могут быть лучше и проще.

## 4. “Потоки — это то же самое, что async”

Нет.  
`threading` — это системные потоки.  
`asyncio` — это кооперативная многозадачность в одном потоке через event loop. Это отдельная тема.

---

# Как можно кратко ответить на аттестации

Можно дать такой сжатый ответ:

`multithreading` — это несколько потоков внутри одного процесса. Потоки разделяют память, поэтому обмен данными между ними дешёвый, но возникает риск гонок. В CPython потоки ограничены GIL, поэтому для CPU-bound задач они обычно не дают ускорения, зато хорошо подходят для I/O-bound операций, например сети и файлов.

`multiprocessing` — это несколько отдельных процессов. У каждого процесса своё адресное пространство и свой интерпретатор Python, поэтому можно добиться реального параллелизма на нескольких ядрах и обойти GIL. Но процессы дороже создавать, они потребляют больше памяти, и для обмена данными между ними нужен IPC, часто через pickle и очереди.

---

# Короткое сравнение в одной фразе

**Threads** — дешевле, разделяют память, хороши для I/O.  
**Processes** — тяжелее, память раздельная, хороши для CPU.

# Мини-таблица для запоминания

|Критерий|Multithreading|Multiprocessing|
|---|---|---|
|Единица выполнения|Поток|Процесс|
|Память|Общая|Раздельная|
|Создание|Дешевле|Дороже|
|Обмен данными|Проще|Сложнее|
|GIL|Ограничивает CPU-bound|Обходится|
|I/O-bound|Отлично|Можно, но часто избыточно|
|CPU-bound|Обычно плохо в CPython|Обычно хорошо|

# Что можно сказать в конце как вывод

Итог такой: выбор между multithreading и multiprocessing зависит не от моды, а от характера нагрузки. Если задача упирается в ожидание внешних ресурсов — чаще подходят потоки. Если задача упирается в вычисления на CPU — чаще подходят процессы. В Python это особенно важно из-за GIL, который делает distinction между I/O-bound и CPU-bound задачами принципиальным.

# Вопросы, которые могут задать

## 1. Что такое GIL и почему он важен?

GIL — это глобальная блокировка интерпретатора в CPython. Она не даёт нескольким потокам одновременно исполнять Python bytecode в одном процессе. Поэтому threads подходят для I/O, но не очень подходят для CPU-bound задач.

## 2. Почему потоки всё-таки полезны, если есть GIL?

Потому что во время ожидания I/O поток может освободить GIL, и другой поток будет выполняться.

## 3. Почему multiprocessing обходит GIL?

Потому что каждый процесс имеет свой собственный интерпретатор Python и свой GIL.

## 4. Что дороже: поток или процесс?

Процесс дороже по запуску и по памяти. Поток легче.

## 5. Почему обмен данными между процессами дороже?

Потому что память не общая, и данные нужно передавать через IPC, часто с сериализацией.

## 6. Когда использовать multithreading?

Когда задача I/O-bound: сеть, файлы, API, ожидание БД.

## 7. Когда использовать multiprocessing?

Когда задача CPU-bound: расчёты, тяжёлый парсинг, трансформации на чистом Python.

## 8. Какие риски у multithreading?

Race condition, deadlock, проблемы синхронизации, порча общего состояния.

## 9. Какие риски у multiprocessing?

Высокий overhead, расход памяти, стоимость сериализации, сложность обмена данными.

## 10. Чем threading отличается от asyncio?

`threading` — это реальные потоки ОС.  
`asyncio` — это кооперативная многозадачность в одном потоке через event loop.

## 11. Можно ли использовать multiprocessing для работы с большими pandas DataFrame?

Можно, но нужно учитывать стоимость копирования и сериализации. Иногда это съедает выигрыш.

## 12. Что такое race condition?

Это ситуация, когда результат зависит от порядка выполнения потоков, и из-за конкурентного доступа к общим данным поведение становится нестабильным.

## 13. Что такое deadlock?

Это взаимная блокировка, когда несколько потоков или процессов ждут друг друга и никто не может продолжить выполнение.

## 14. Почему в Windows с multiprocessing нужен `if __name__ == "__main__"`?

Потому что там часто используется `spawn`, и модуль импортируется заново в дочернем процессе. Без этой защиты можно случайно рекурсивно пересоздавать процессы.
