## Доклад

ORM — это **Object-Relational Mapping**, то есть механизм, который связывает объекты в Python с таблицами в реляционной базе данных.

Если говорить просто, ORM нужен для того, чтобы программист работал не только через SQL-запросы, но и через Python-объекты.  
Например, у нас есть таблица `users`, и ей может соответствовать класс `User`. Тогда строка в таблице — это объект `User`, а колонки таблицы — это поля объекта.

Например, если есть таблица:
```python
users(id, name, email)
```
то в Python ей может соответствовать объект:
``` python
user = User(id=1, name="Alice", email="alice@example.com")
```
---

## Что делает ORM под капотом

Самое важное здесь понять вот что:  
**ORM не отменяет SQL.**  
Под капотом всё равно выполняются обычные запросы:

- `SELECT`
- `INSERT`
- `UPDATE`
- `DELETE`

То есть ORM — это не “магия вместо базы”, а **прослойка**, которая помогает работать с базой удобнее.

Например, если мы в Python создаём объект:
``` python
user = User(name="Alice", email="alice@example.com")
```
а потом сохраняем его, то под капотом ORM в какой-то момент всё равно сформирует что-то вроде:
``` python
INSERT INTO users (name, email) VALUES ('Alice', 'alice@example.com');
```
Только сделает это сам, без того чтобы мы руками писали SQL в каждом месте.

---

## Базовая идея ORM

ORM решает две задачи:

### Первая задача

Понять, **как объект связать с таблицей**.

Например:

- класс `User` → таблица `users`
- поле `name` → колонка `name`
- поле `email` → колонка `email`

### Вторая задача

Понять, **какой SQL нужно выполнить**.

Например:

- если объект новый — сделать `INSERT`
- если объект уже существует — сделать `UPDATE`
- если нужно прочитать данные — сделать `SELECT`

---

# Основные подходы ORM

Когда говорят про **основные подходы ORM**, обычно имеют в виду **не виды SQL**, а то, **где находится логика работы с базой**.

Проще говоря, вопрос такой:

**Кто именно отвечает за сохранение объекта в БД?**

И здесь обычно выделяют три вещи, которые полезно понимать:

1. **Active Record**
2. **Data Mapper**
3. **Repository / DAO** — это уже скорее архитектурный слой поверх ORM, но на практике его тоже часто обсуждают вместе с основными подходами.

---

# 1. Active Record

## Идея

В подходе **Active Record** сам объект умеет себя сохранять, обновлять, удалять и иногда загружать.

То есть модель одновременно содержит:

- данные;
- и логику работы с БД.

Например:
```python
user = User(name="Alice", email="alice@example.com")  
user.save()
```
Здесь ключевая идея в том, что **объект сам знает, как себя сохранить**.

---

## Что происходит под капотом

Допустим, мы вызываем:
```python
user.save()
```
Дальше логика примерно такая:

### Шаг 1

Метод `.save()` смотрит, есть ли у объекта `id`.

### Шаг 2

Если `id` нет, значит объект новый, и нужен `INSERT`.

### Шаг 3

Если `id` есть, значит запись уже существует, и нужен `UPDATE`.

### Шаг 4

Модель сама формирует SQL.

Например, для нового объекта:
```SQL
INSERT INTO users (name, email) VALUES (%s, %s)
```
### Шаг 5

Этот SQL отправляется в БД.

### Шаг 6

Если база вернула новый `id`, он записывается обратно в объект.

---

## Как это проговорить просто

**Active Record — это подход, где модель сама умеет работать с базой. Мы вызываем у объекта `save()`, и он сам решает, делать `INSERT` или `UPDATE`.**

---

## Плюсы

- просто понять;
- быстро писать CRUD;
- удобно для небольших приложений;
- меньше инфраструктурного кода.

## Минусы

- модель знает слишком много;
- смешиваются данные и доступ к БД;
- на сложных проектах архитектура может становиться грязнее.

---

# 2. Data Mapper

## Идея

В подходе **Data Mapper** объект **не умеет сохранять себя сам**.  
Он просто хранит данные.  
А за работу с базой отвечает **отдельный слой** — mapper или session.

То есть здесь логика уже другая:
```python
user = User(name="Alice", email="alice@example.com")  
mapper.save(user)
```
или, если говорить ближе к SQLAlchemy:
```python
session.add(user)  
session.commit()
```
Главная мысль:  
**объект не знает деталей SQL и не занимается сохранением напрямую**.

---

## Что происходит под капотом

Допустим, мы пишем:
```python
session.add(user)  
session.commit()
```
Тогда упрощённо происходит следующее:

### Шаг 1

Создаётся объект `User` в памяти.

### Шаг 2

`session.add(user)` помечает его как новый объект.

### Шаг 3

При `commit()` session смотрит, какие объекты:

- новые,
- изменённые,
- удалённые.

### Шаг 4

Для новых объектов генерируется `INSERT`.

### Шаг 5

Для изменённых объектов — `UPDATE`.

### Шаг 6

Все запросы отправляются в БД в рамках транзакции.

### Шаг 7

После успешного выполнения транзакция коммитится.

---

## Как это проговорить просто

**Data Mapper — это подход, где модель отделена от работы с БД. Объект просто хранит данные, а сохранением управляет отдельный слой, например mapper или session.**

---

## Плюсы

- чище архитектура;
- лучше разделение ответственности;
- удобнее для больших и сложных систем;
- проще контролировать транзакции и слой доступа к данным.

## Минусы

- сложнее понять с нуля;
- больше абстракций;
- больше служебного кода.

---

# 3. Repository / DAO

Вот здесь как раз была главная путаница, поэтому объясню особенно аккуратно.

## Идея

**Repository** — это не совсем “ещё один тип маппера”.  
Это **более высокий слой**, через который приложение работает с данными.

То есть приложение говорит не:
```python
mapper.insert(user)
```
а скорее:
```python
user_repository.save(user)  
user_repository.get_by_email(email)
```
---

## В чём его смысл

Repository нужен для того, чтобы **спрятать детали хранения данных** от остального приложения.

Бизнес-код не должен знать:

- где именно лежат данные;
- какой SQL используется;
- как устроен mapper;
- есть ли внутри ORM, raw SQL или что-то ещё.

Он должен знать только, что есть, например, `UserRepository`, у которого есть методы:

- `save(user)`
- `get_by_id(id)`
- `get_by_email(email)`

---

## Чем Repository отличается от Mapper

Это самый важный момент.

### Mapper

Отвечает за вопрос:

**Как объект превратить в запись таблицы и обратно?**

То есть mapper — это по сути **переводчик** между объектом и строкой БД.

Он думает про:

- таблицы;
- колонки;
- `INSERT`, `UPDATE`;
- преобразование `row -> object`.

### Repository

Отвечает за вопрос:

**Как приложению удобно получать и сохранять сущности?**

Он думает уже не про колонки, а про сценарии уровня приложения:

- получить пользователя по email;
- сохранить пользователя;
- получить активных пользователей;
- получить клиентов с просроченными заказами.

---

## Почему они кажутся одинаковыми

Потому что в маленьких примерах и repository, и mapper часто выглядят как обычный класс с SQL внутри.

И это правда:  
на небольшом проекте один класс может одновременно делать и repository-роль, и mapper-роль.

Но **архитектурно** это разные обязанности:

- **mapper** — низкий уровень, преобразование объекта и строки таблицы;
- **repository** — высокий уровень, интерфейс доступа к данным для приложения.

---

## Как проговорить это внятно

**Mapper отвечает за техническое преобразование объекта в запись БД и обратно. Repository — это более высокий слой, через который приложение получает и сохраняет сущности, не зная деталей SQL и ORM.**

---

# Очень простой пример на одной сущности

Пусть есть таблица `users`.

## Active Record
```python
user = User(name="Alice", email="alice@example.com")  
user.save()
```
Смысл:  
**объект сам себя сохраняет**.

---

## Data Mapper
```python
user = User(name="Alice", email="alice@example.com")  
mapper.save(user)
```
Смысл:  
**объект не сохраняет себя сам, это делает отдельный mapper**.

---

## Repository
```python
user = User(name="Alice", email="alice@example.com")  
user_repository.save(user)
```
Смысл:  
**приложение работает не с SQL и не с mapper напрямую, а с репозиторием как с интерфейсом доступа к данным**.

---

# Как это связать вместе

Можно представить три уровня понимания:

### Уровень 1 — объект

Например, `User(name="Alice")`

### Уровень 2 — маппинг

Кто-то должен понять:

- в какую таблицу писать;
- какие колонки заполнять;
- какой сделать SQL.

Это и есть логика **mapper**.

### Уровень 3 — доступ для приложения

Приложению удобнее работать не с SQL, а с чем-то вроде:

- `get_user_by_email`
- `save_user`
- `get_active_users`

Это уже роль **repository**.

---

# Где что обычно встречается в Python

## Django ORM

Обычно объясняют как подход, близкий к **Active Record**, потому что модель часто умеет:
```python
user.save()  
user.delete()
```
## SQLAlchemy

Обычно ближе к **Data Mapper**, потому что там явно есть session:
```python
session.add(user)  
session.commit()
```
## Repository

Часто добавляют поверх SQLAlchemy или другого ORM, если хотят более чистую архитектуру.

---

# Что важно сказать именно для Data Engineer

Для Data Engineer ORM — это полезный инструмент, но не универсальная замена SQL.

Он удобен:

- для служебных таблиц;
- для конфигураций;
- для статусов задач;
- для небольших внутренних сервисов;
- для простого CRUD.

Но для сложной аналитики, массовых загрузок и тяжёлых ETL-процессов чаще нужен чистый SQL или комбинированный подход.

То есть сильный ответ здесь такой:  
**ORM удобен там, где нужно работать с сущностями и типовыми операциями, но он не отменяет необходимости понимать SQL и то, что происходит под капотом.**

---

# Короткий вывод

Итого:  
ORM — это слой, который связывает Python-объекты и таблицы базы данных.  
Под капотом ORM всё равно генерирует обычный SQL, но делает это автоматически.

Основные подходы можно объяснить так:

- **Active Record** — объект сам умеет работать с БД;
- **Data Mapper** — объект отделён от механизма сохранения, и этим занимается отдельный слой;
- **Repository** — более высокий слой, который даёт приложению удобный интерфейс работы с данными и может использовать mapper, ORM или SQL внутри.

В Active Record сама модель умеет себя сохранять, например через `save()`. В Data Mapper модель отделена от работы с БД, а сохранением управляет отдельный слой, например session. # Вопросы, которые могут задать

### 1. Что такое ORM?

Это Object-Relational Mapping, то есть сопоставление объектов программы и таблиц реляционной базы данных.

### 2. ORM заменяет SQL?

Нет. Под капотом ORM всё равно генерирует и выполняет обычный SQL.

### 3. Что такое Active Record?

Это подход, где модель сама умеет себя сохранять, обновлять и удалять.

### 4. Что такое Data Mapper?

Это подход, где модель отделена от механизма работы с БД, а сохранением занимается отдельный слой.

### 5. В чём главное отличие Active Record от Data Mapper?

В Active Record объект сам работает с БД.  
В Data Mapper объект только хранит данные, а SQL и сохранение делает внешний слой.

### 6. Что такое Repository?

Это слой доступа к данным, через который приложение получает и сохраняет сущности, не зная деталей SQL и ORM.

### 7. Чем Repository отличается от Mapper?

Mapper отвечает за преобразование объекта в запись БД и обратно.  
Repository отвечает за удобный интерфейс работы с данными для приложения.

### 8. Почему Repository и Mapper кажутся похожими?

Потому что в маленьких примерах оба могут содержать SQL. Но mapper — это низкоуровневый переводчик, а repository — более высокий интерфейс доступа к данным.

### 9. Что ближе к Active Record в Python?

Обычно Django ORM.

### 10. Что ближе к Data Mapper в Python?

Обычно SQLAlchemy.

### 11. Где ORM полезен для Data Engineer?

В служебных таблицах, внутренних сервисах, конфигурациях, статусах задач и CRUD-операциях.

### 12. Где ORM не лучший выбор?

В сложной аналитике, тяжёлых ETL, массовых загрузках и там, где нужен полный контроль над SQL.
