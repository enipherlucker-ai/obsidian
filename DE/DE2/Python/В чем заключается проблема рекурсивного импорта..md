### 7.1. Что такое циклический импорт

**Циклический (рекурсивный) импорт** возникает, когда модуль `A` импортирует модуль `B`, а модуль `B` (прямо или транзитивно) импортирует модуль `A`.

```
A.py  ──import──▶  B.py
 ▲                   │
 └───── import ──────┘
```

### 7.2. Как работает механизм импорта в Python

1. При `import module_x` Python проверяет кэш `sys.modules`.
2. Если модуль **уже есть** в кэше — возвращает кэшированный объект модуля (даже если он ещё не до конца инициализирован!).
3. Если модуля **нет** — создаёт пустой объект `ModuleType`, помещает его в `sys.modules`, затем **выполняет код файла сверху вниз**.
4. По окончании выполнения модуль считается полностью инициализированным.

**Ключевой момент:** на шаге 2 модуль может быть **частично инициализирован** — те имена, определение которых ещё не выполнено, в нём **отсутствуют**.

### 7.3. Конкретный пример сбоя

```python
# ——— models.py ———
from services import get_user_service   # ← при импорте services.py начнёт выполняться

class User:
    def __init__(self, name: str):
        self.name = name
```

```python
# ——— services.py ———
from models import User                # ← при импорте models.py (уже в sys.modules, но User ещё не определён!)

def get_user_service() -> User:
    return User("Alice")
```

**Что происходит:**

1. Python начинает выполнять `models.py`.
2. Первая строка — `from services import get_user_service` → Python начинает выполнять `services.py`.
3. В `services.py` первая строка — `from models import User` → Python смотрит в `sys.modules`: `models` **уже там** (шаг 1), но класс `User` ещё **не определён** (выполнение `models.py` остановилось на строке 1).
4. Результат: `ImportError: cannot import name 'User' from partially initialized module 'models'`.

### 7.4. Важный нюанс: `import module` vs `from module import name`

```python
# Этот вариант ЧАЩЕ работает при циклах:
import models          # получаем ссылку на объект модуля (даже частично инициализированный)
# ...
models.User            # атрибут ищется в момент обращения, а не импорта

# Этот вариант ЧАЩЕ ломается:
from models import User  # пытаемся достать User СЕЙЧАС — его ещё нет → ImportError
```

`import module` откладывает разрешение имени до момента использования, поэтому к тому времени модуль может быть уже полностью инициализирован. Однако это **хрупкое** решение — лучше устранить цикл.

### 7.5. Все способы решения

#### 7.5.1. Ленивый импорт внутри функции

Импорт выполняется только при вызове функции, когда оба модуля уже полностью загружены.

```python
# services.py
def get_user_service():
    from models import User   # импорт откладывается до вызова функции
    return User("Alice")
```

**Плюс:** минимальные изменения.
**Минус:** импорт скрыт от IDE и линтеров, снижает читаемость.

#### 7.5.2. Вынесение общего кода в третий модуль

Разорвать цикл, выделив общую зависимость.

```
До:   A ↔ B
После: A → C ← B   (C не зависит ни от A, ни от B)
```

```python
# base.py (новый модуль)
class User:
    def __init__(self, name: str):
        self.name = name

# models.py
from base import User   # нет цикла

# services.py
from base import User   # нет цикла
```

**Плюс:** чистая архитектура.
**Минус:** требует рефакторинга.

#### 7.5.3. `TYPE_CHECKING` (только для аннотаций типов)

Если импорт нужен исключительно для type hints, можно использовать блок `if TYPE_CHECKING`, который выполняется **только** статическими анализаторами (mypy, pyright), но **не** во время работы программы.

```python
from __future__ import annotations   # все аннотации становятся строками (PEP 563)
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from models import User          # не выполняется в runtime

def get_user_service() -> "User":    # строковая аннотация (forward reference)
    from models import User          # реальный импорт для runtime
    return User("Alice")
```

#### 7.5.4. Dependency Injection (внедрение зависимости)

Вместо импорта конкретного класса модуль принимает зависимость через параметр.

```python
# services.py — не импортирует models вообще
def get_user_service(user_class):
    return user_class("Alice")

# main.py
from models import User
from services import get_user_service

user = get_user_service(User)
```

#### 7.5.5. Абстрактные базовые классы (ABC) / протоколы

Оба модуля зависят от абстракции, а не друг от друга (Dependency Inversion Principle).

```python
# interfaces.py
from abc import ABC, abstractmethod

class IUser(ABC):
    @abstractmethod
    def get_name(self) -> str: ...

# models.py
from interfaces import IUser

class User(IUser):
    def __init__(self, name: str):
        self.name = name
    def get_name(self) -> str:
        return self.name

# services.py
from interfaces import IUser

def get_user_service(user: IUser) -> str:
    return user.get_name()
```

### 7.6. Как обнаружить циклические импорты

| Способ | Описание |
|--------|----------|
| `python -v script.py` | Включает вербозный вывод: видно порядок загрузки модулей |
| Traceback | `ImportError: cannot import name 'X' from partially initialized module` — прямое указание |
| `pylint` / `flake8-import-order` | Статические анализаторы предупреждают о подозрительных зависимостях |
| `pydeps`, `import-linter` | Визуализация графа зависимостей, обнаружение циклов |
| `sys.modules` отладка | Проверить состояние модуля: `print(dir(sys.modules['models']))` — увидеть, какие имена уже определены |

### 7.7. Резюме

Циклический импорт — это **архитектурная проблема**, сигнализирующая о нарушении направленности зависимостей. Лучший способ борьбы — **рефакторинг** (вынесение в третий модуль, DI, абстракции). Ленивый импорт внутри функции — допустимый, но менее чистый workaround.
