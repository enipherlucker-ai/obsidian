## 12. Обработка исключений.

### 12.1. Полный синтаксис try/except/else/finally

```python
try:
    # Код, который может вызвать исключение
    result = 10 / x
except ZeroDivisionError as e:
    # Обработка конкретного типа исключения
    print(f"Деление на ноль: {e}")
except (ValueError, TypeError) as e:
    # Ловим несколько типов одновременно
    print(f"Ошибка значения/типа: {e}")
except Exception as e:
    # Ловим все «обычные» исключения (fallback)
    print(f"Непредвиденная ошибка: {e}")
else:
    # Выполняется, ТОЛЬКО если в try НЕ было исключений
    print(f"Результат: {result}")
finally:
    # Выполняется ВСЕГДА — и при ошибке, и без неё, и при return
    print("Очистка ресурсов")
```

| Блок      | Когда выполняется                                 | Назначение                                                            |
| --------- | ------------------------------------------------- | --------------------------------------------------------------------- |
| `try`     | Всегда                                            | Основной код, который может упасть                                    |
| `except`  | Если в `try` возникло подходящее исключение       | Обработка/восстановление после ошибки                                 |
| `else`    | Если в `try` НЕ было исключений                   | Код, который должен выполниться только при успехе (не защищён except) |
| `finally` | ВСЕГДА (даже при `return`, `break`, `sys.exit()`) | Гарантированная очистка (закрытие файлов, соединений)                 |

### 12.2. Иерархия исключений

```
BaseException
├── SystemExit              # sys.exit()
├── KeyboardInterrupt       # Ctrl+C
├── GeneratorExit           # generator.close()
└── Exception               # базовый класс «обычных» исключений
    ├── StopIteration
    ├── ArithmeticError
    │   ├── ZeroDivisionError
    │   ├── OverflowError
    │   └── FloatingPointError
    ├── LookupError
    │   ├── IndexError
    │   └── KeyError
    ├── AttributeError
    ├── TypeError
    ├── ValueError
    │   └── UnicodeError
    ├── OSError (IOError)
    │   ├── FileNotFoundError
    │   ├── PermissionError
    │   └── ConnectionError
    │       ├── ConnectionRefusedError
    │       └── ConnectionResetError
    ├── RuntimeError
    │   └── RecursionError
    └── ...
```

**Важно:** `except Exception` ловит все «нормальные» ошибки, но **не** ловит `SystemExit`, `KeyboardInterrupt`, `GeneratorExit` — они наследуются от `BaseException` напрямую.

### 12.3. Голый `except` — почему это плохо

```python
# ПЛОХО — ловит ВСЁ, включая KeyboardInterrupt и SystemExit
try:
    do_something()
except:
    pass  # «проглатываем» ошибку — невозможно отладить

# ПРАВИЛЬНО — ловим конкретные типы
try:
    do_something()
except (ValueError, IOError) as e:
    logger.error("Ошибка: %s", e)
```

### 12.4. `raise` и повторный вызов исключения

```python
# Повторный вызов текущего исключения (сохраняет traceback)
try:
    process_data()
except ValueError:
    logger.error("Ошибка обработки данных")
    raise  # ← пробрасываем дальше, traceback не теряется

# Вызов нового исключения
def validate_age(age):
    if age < 0:
        raise ValueError(f"Возраст не может быть отрицательным: {age}")
    if age > 150:
        raise ValueError(f"Нереалистичный возраст: {age}")
```

### 12.5. Цепочки исключений (Exception Chaining)

```python
# Явная цепочка: raise X from Y
# Устанавливает __cause__, указывая на первоначальную ошибку
try:
    value = int(user_input)
except ValueError as e:
    raise ValidationError(f"Невалидный ввод: {user_input}") from e
# Traceback покажет:
# ValueError: invalid literal for int()...
# The above exception was the direct cause of the following exception:
# ValidationError: Невалидный ввод: abc

# Неявная цепочка: __context__
# Если в except возникает новое исключение без from,
# Python автоматически сохраняет оригинал в __context__
try:
    do_something()
except TypeError:
    raise RuntimeError("Something went wrong")
# Traceback: "During handling of the above exception, another exception occurred"

# Подавление цепочки: raise X from None
try:
    do_something()
except TypeError:
    raise RuntimeError("Clean error") from None
# Traceback покажет только RuntimeError
```

### 12.6. Создание пользовательских исключений

```python
class AppError(Exception):
    """Базовое исключение приложения."""
    pass

class NotFoundError(AppError):
    """Ресурс не найден."""
    def __init__(self, resource: str, resource_id: int):
        self.resource = resource
        self.resource_id = resource_id
        super().__init__(f"{resource} с id={resource_id} не найден")

class ValidationError(AppError):
    """Ошибка валидации."""
    def __init__(self, field: str, message: str):
        self.field = field
        self.message = message
        super().__init__(f"Валидация поля '{field}': {message}")

# Использование
try:
    user = get_user(user_id=999)
    if user is None:
        raise NotFoundError("User", 999)
except NotFoundError as e:
    print(e)                # User с id=999 не найден
    print(e.resource)       # User
    print(e.resource_id)    # 999
```

### 12.7. Контекстные менеджеры (`with`)

Контекстный менеджер гарантирует выполнение кода очистки, даже если возникло исключение.

**Протокол:**

```python
class ManagedResource:
    def __enter__(self):
        # Инициализация / захват ресурса
        print("Открытие ресурса")
        return self  # возвращается в as-переменную

    def __exit__(self, exc_type, exc_val, exc_tb):
        # Освобождение ресурса (ВСЕГДА вызывается)
        print("Закрытие ресурса")
        # Если return True — исключение подавляется
        # Если return False/None — исключение пробрасывается
        return False

with ManagedResource() as res:
    print("Работа с ресурсом")
    # Даже при исключении __exit__ будет вызван
```

**`contextlib.contextmanager` — генератор вместо класса:**

```python
from contextlib import contextmanager
import time

@contextmanager
def timer(label):
    start = time.perf_counter()
    try:
        yield  # здесь выполняется тело with-блока
    finally:
        elapsed = time.perf_counter() - start
        print(f"{label}: {elapsed:.3f} сек")

with timer("Обработка данных"):
    data = [i ** 2 for i in range(1_000_000)]
```

**Вложенные контекстные менеджеры:**

```python
# Несколько менеджеров в одном with
with open("input.txt") as fin, open("output.txt", "w") as fout:
    for line in fin:
        fout.write(line.upper())

# Или через contextlib.ExitStack (динамическое количество)
from contextlib import ExitStack

with ExitStack() as stack:
    files = [stack.enter_context(open(f"file_{i}.txt")) for i in range(10)]
```

### 12.8. LBYL vs EAFP

**LBYL** (Look Before You Leap) — проверяй перед действием:

```python
# LBYL-стиль
if key in my_dict:
    value = my_dict[key]
else:
    value = default

if os.path.exists(filepath):
    with open(filepath) as f:
        data = f.read()
```

**EAFP** (Easier to Ask Forgiveness than Permission) — делай, лови исключение:

```python
# EAFP-стиль (идиоматичный Python)
try:
    value = my_dict[key]
except KeyError:
    value = default

try:
    with open(filepath) as f:
        data = f.read()
except FileNotFoundError:
    data = None
```

**В Python принят EAFP-стиль**, потому что:
- Избегает race condition (файл может исчезнуть между `exists()` и `open()`).
- Быстрее в «happy path» (нет двойного обращения: проверка + действие).
- Но `try/except` медленнее при частых ошибках.

### 12.9. Логирование исключений

```python
import logging

logger = logging.getLogger(__name__)

try:
    process_data()
except Exception as e:
    # logger.exception() — логирует с уровнем ERROR + полный traceback
    logger.exception("Ошибка обработки данных")

    # logger.error() — логирует только сообщение (без traceback)
    logger.error("Ошибка: %s", e)

    # logger.error() с traceback (явно)
    logger.error("Ошибка: %s", e, exc_info=True)
```

**Разница:**
- `logger.exception(msg)` ≡ `logger.error(msg, exc_info=True)` — всегда выводит traceback.
- `logger.error(msg)` — только текст сообщения.

### 12.10. Группы исключений (Python 3.11+)

Когда нужно выбросить **несколько** исключений одновременно (например, параллельные задачи `asyncio.TaskGroup`):

```python
# ExceptionGroup — группа исключений
eg = ExceptionGroup("ошибки валидации", [
    ValueError("поле 'name' пустое"),
    TypeError("поле 'age' — не число"),
    ValueError("поле 'email' невалидно"),
])
raise eg

# Ловим с помощью except* (звёздочка!)
try:
    raise ExceptionGroup("batch errors", [
        ValueError("bad value"),
        TypeError("bad type"),
        OSError("file not found"),
    ])
except* ValueError as eg:
    print(f"ValueError(s): {eg.exceptions}")  # ловит оба ValueError
except* TypeError as eg:
    print(f"TypeError(s): {eg.exceptions}")
except* OSError as eg:
    print(f"OSError(s): {eg.exceptions}")
```

**Ключевые особенности `except*`:**
- `except*` может поймать **подмножество** исключений из группы.
- Несколько `except*`-блоков могут сработать для одной `ExceptionGroup`.
- Полезно для `asyncio.TaskGroup`, который собирает ошибки из параллельных задач.

```python
import asyncio

async def failing_task(n):
    raise ValueError(f"Task {n} failed")

async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(failing_task(1))
            tg.create_task(failing_task(2))
    except* ValueError as eg:
        for exc in eg.exceptions:
            print(f"Caught: {exc}")

asyncio.run(main())
```
